## 메뉴 재구성 — 환자 중심 워크플로우

### 기존 → 변경
```
기존 (기능 중심, 10개 메뉴):
  1.신환등록 2.브리핑 3.차트입력 4.환자목록 5.검색
  6.수정 7.삭제 8.통계 9.연구검색 10.끝

변경 (환자 중심, 4개 메뉴):
  1. 신환등록
  2. 환자목록
     1. 전체 환자 목록
     2. 환자 검색 → 환자 선택 → 환자 메뉴:
        1. 브리핑
        2. 진료 기록 작성
        3. 수정
        4. 삭제
        5. 전체 이력 보기
  3. 연구
     1. 전체 통계
     2. 연구 도구 (자연어 검색, AI 자동 통계, 단계별 통계)
  4. 끝
  (admin 히든 유지)
```

### 설계 원칙
```
기존: 기능 선택 → 환자 선택 (매번 다시 선택해야 함)
변경: 환자 선택 → 해당 환자에 대해 여러 작업 가능

실제 진료 흐름과 일치:
  환자 검색 → 브리핑 확인 → 차트 작성 → 수정/삭제 필요 시

"돌아가기" 네비게이션: Week 5~6 프론트에서 구현
  → 지금 터미널에서는 기능 실행 후 메인 메뉴로 복귀
  → 나중에 프론트에서 환자 메뉴에 머물기 + 돌아가기 버튼

기술 용어: "워크플로우 리팩토링(workflow refactoring)"
  → 화면은 안 바꾸고 업무 흐름만 재설계하는 것
```

## 의무기록 수정/삭제 정책 — 백엔드 구현 완료

### 핵심 원칙
```
의무기록 법적 원칙:
  - 한 번 작성된 기록은 변경 불가 (법적 증거)
  - 수정 = 원본 보존 + 정정 기록 추가 (덮어쓰기 금지!)
  - 삭제 = 실제 삭제 안 함 + 무효 플래그 처리
  - AI가 이후 차트를 자동 수정하면 안 됨 (의사가 직접!)
```

### 유효여부 플래그
```
모든 데이터 테이블에 추가:
  유효여부 INTEGER DEFAULT 1  (1=유효, 0=무효)
  → 검사결과, 영상검사, 진단, 추적계획, 방문

진료/브리핑: 유효여부=1만 조회
통계:       유효여부=1만 조회 → 오기입 값 자동 제외
이력 보기:  전부 보임 (정정 이력 포함)
```

### 수정 흐름
```
의사: "260301 LDL 173을 163으로 정정"
  ↓
시스템:
  1. 기존 검사id=1 → 유효여부=0
  2. 새 검사결과 추가: LDL 163, 유효여부=1
  3. free_text에 정정 기록 추가
  ↓
AI가 영향받는 이후 차트 검색 → 의사에게 제시
  ↓
의사가 각 차트를 열어서 직접 수정 → 같은 패턴 반복
  ↓
더 이상 영향 없으면 끝
```

### 삭제 흐름
```
의사: "260301 기록 삭제"
  ↓
시스템:
  1. 방문 유효여부=0 (원본 보관)
  2. 해당 방문의 진단/검사/영상/추적도 유효여부=0
  3. 무효 사유 기록
  ↓
AI가 영향받는 이후 차트 검색 → 의사에게 제시
  ↓
의사가 직접 처리 → 같은 패턴 반복
  ↓
더 이상 영향 없으면 끝
```

### 통계에서 무결성 보장
```
문제: 오기입 값(173)과 정정값(163)이 동시에 존재하면 통계 왜곡
해결: WHERE 유효여부=1 → 정정값(163)만 포함, 오기입(173) 자동 제외
```

### 구현된 새 함수들
```
util.py:
  ✅ 모든 수정 함수 → 원본 무효 + 새 레코드 방식으로 변경
  ✅ 모든 삭제 함수 → DELETE 대신 유효여부=0 UPDATE
  ✅ 모든 조회 함수 → WHERE 유효여부=1 추가
  ✅ 전체이력조회_무효포함(환자id) — 무효 기록 포함 전체 조회
  ✅ 정정이력조회(환자id) — 정정 이력만 조회
  ✅ 영향받는_차트_검색(환자id, 검색어, 기준방문일) — 이후 차트 안내
  ※ 환자삭제만 예외적 실제 DELETE 유지 (테스트 데이터 정리용)
```

## free_text AI 분석 기능 — 모드 1, 2에 통합

### 별도 메뉴가 아니라 자동 판단
```
AI가 연구 질문을 보고 free_text 분석이 필요한지 자동 판단:

필요한 경우:
  "부작용 호소한 환자 분석" → 증상 표현이 다양 (근육통, myalgia, 뻣뻣함...)
  "금연 성공 패턴" → 경과 기록 분석 필요

불필요한 경우:
  "LDL 평균 구해줘" → 검사결과 테이블만으로 충분
  "고혈압 환자 수" → 진단 테이블로 충분
```

### SQL 선추림 vs 전체 분석
```
1번 (SQL 선추림 — 권장):
  SQL로 후보 환자 추림 → 추려진 환자의 free_text만 AI 분석
  비용: 1/4 ~ 1/20
  정확도: 95~99%

2번 (전체 분석):
  모든 환자의 free_text를 AI 분석
  비용: 높음
  정확도: 100%
  → 논문 발표용 정밀 분석 시 사용
```

### 비식별화 + 비용 안내
```
AI에게 보낼 때:
  이름 → 환자001, 환자002
  생년월일 → 나이대

결과 출력 시:
  비식별 → 실명 복원

비용 안내:
  "분석 대상: 30명, free_text 150건
   예상 토큰: 약 37K
   진행할까요? (y/n)"
```

## 설계 토론 — 의무기록 원칙

### "free_text만 저장하면 안 되나?" 질문에 대한 결론
```
free_text만 저장 시:
  ❌ 통계할 때마다 AI가 전체 free_text 재분석 → 비용 폭발
  ❌ 분석할 때마다 결과가 미세하게 달라질 수 있음
  ❌ 브리핑도 매번 전체 재분석 필요

현재 구조 (정형+비정형 병행):
  ✅ 입력 시 한 번만 분석 → SQL에 즉시 저장
  ✅ 검색/통계가 SQL로 즉시 가능
  ✅ free_text 원본도 보존
  → 구조 변경 필요 없음!
  → 수정/삭제 정책만 바꾸면 됨 (유효여부 플래그)
```

### "이후 기록에 영향이 가면?" 질문에 대한 결론
```
이후 기록을 AI가 자동 수정하면:
  ❌ 의사가 작성하지 않은 기록이 됨 → 의무기록 위반
  ❌ 사용자도 모르는 사이에 기록이 바뀜

올바른 방식:
  ✅ AI는 "영향받는 차트 목록"만 찾아줌
  ✅ 의사가 직접 하나씩 수정 → 법적으로 유효
  ✅ 수정할 때마다 같은 패턴 반복 (처리→검색→수정→반복)
```
