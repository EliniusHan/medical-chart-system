## 설계 심화: 데이터 흐름과 수정/삭제 로직

### 핵심 논의 — 입력과 수정/삭제의 방향이 다르다

**입력할 때 (한 방향으로 뿌림):**
```
의사가 free-text 씀 → AI가 분석 → 각 테이블로 분배 (한꺼번에)
```

**저장 후에는 각각 독립적:**
```
방문 free_text = "그날의 기록" → 삭제하면 그날 기록만 사라짐
진단 = "환자의 상태" → 차트 하나 지운다고 없어지면 안 됨
검사결과 = "객관적 수치" → 차트 하나 지운다고 사라지면 안 됨
추적계획 = "앞으로의 계획" → 차트 하나 지운다고 없어지면 안 됨
```

**비유: 차트 한 장을 찢어도 환자의 고혈압 진단이 취소되진 않음**

---

### 검사결과 vs 진단 — 성격이 완전히 다름

**검사결과 (객관적 사실, 불변):**
```
FBS 124 → 영원히 124
원본 1개만 존재 → 수정하면 전체 반영 → 삭제하면 전체 반영
다른 차트에서 같은 수치를 언급해도 복사 저장하지 않음 (글로 적는 것뿐)
```

**진단 (시간에 따라 변함):**
```
고혈압 의심 → 고혈압 확진 → 이차성 고혈압 (진화함)
수정 시: "이 차트만" vs "같은 진단 전부" 의사에게 확인
삭제보다 종결 권장 (이력 보존)
```

### 진단 종결 vs 삭제의 차이
```
종결 + 추가 (이력 보존):
  진단id=1, "고혈압 의심", 상태="종결" ← 기록 남음
  진단id=2, "고혈압 확진", 상태="활성" ← 새로 추가
  → 진단 변화 추적 가능 (연구에 활용)
  → 의무기록 원칙: 줄 긋고 옆에 새로 적기

삭제 + 추가 (이력 사라짐):
  진단id=1 삭제됨 → 완전히 사라짐
  진단id=2, "고혈압 확진" ← 새로 추가
  → "의심" 단계가 있었는지 알 수 없음
  → 완전한 오입력일 때만 사용
```

---

### 방문 삭제 시 데이터 보호

```
방문 삭제 시:
  ├── 방문 free_text → 삭제됨
  ├── 검사결과 → 영향 없음 (방문id가 없으니까, 독립적)
  ├── 영상검사 → 영향 없음 (방문id가 없으니까, 독립적)
  ├── 진단 → 유지 (환자에 속한 것)
  └── 추적계획 → 유지 (아직 유효할 수 있음)
```

### 삭제 후 재작성 시 흐름
```
1단계: 방문 기록 삭제 (free_text 사라짐, 테이블 데이터는 남음)
2단계: 새 free_text 작성
3단계: AI가 새 free_text + 기존 테이블 데이터 비교
  - 기존과 동일 → 건너뜀 (자동)
  - 기존과 충돌 → 의사에게 선택지 제시
  - 완전히 새로운 것 → 확인만 받고 추가
4단계: 의사 확인 후 저장
```

---

### free-text 수정 시 테이블 연동

```
의사가 free-text에서 수치 수정 (예: FBS 124 → 128)
  → AI가 변경 감지
  → "검사결과 테이블도 수정할까요?" 확인
  → 승인 시 원본 수정 → 전체 반영

의사가 free-text에서 진단 수정
  → AI가 변경 감지
  → "이 차트만 수정? 같은 진단 전부 수정?" 확인
  → 의사 선택에 따라 처리
```

---

### 이전 검사 기록 관련 흐름

**다음 차트에서 이전 기록을 잘못 입력한 경우:**
```
기존 테이블: FBS 124
의사가 새 차트에 "이전 FBS 121" 적음 (오타)
  → AI: "기존 기록은 124인데 121로 쓰셨네요. 뭐가 맞나요?"
  → 의사 판단에 따라 처리
```

**빠뜨린 기록을 나중에 추가하는 경우:**
```
의사가 C 차트에 "260324 LDL 165" 적음
  → AI: 날짜 인식 → 검사시행일="260324"로 저장
  → 오늘 방문일이 아니라 기록된 날짜로 저장
  (검사결과가 방문과 독립적이라 가능)
```

---

### AI의 데이터 처리 원칙 (3가지 분류)

```
새 데이터       → "이거 넣습니다" (확인만 받고 저장)
충돌 데이터     → "이거 다른데 어떻게 할까요?" (선택 필요)
기존과 동일     → 건너뜀 (자동, 물어보지도 않음)
```

## EMR vs 선생님 시스템의 차이

### 기존 EMR의 한계
```
기존 EMR:
  의사가 free-text 씀 → 그대로 저장 → 끝
  나중에 연구? → 차트 하나하나 열어서 수동 추출 (며칠~몇 주)
```

### 선생님 시스템의 가치
```
선생님 시스템:
  의사가 free-text 씀 → AI가 정형화 → SQL에 구조적 저장
  나중에 연구? → SQL 한 줄로 바로 추출 (몇 초)
```

**이 설계 작업이 나중에 연구할 때 "몇 주를 몇 초로" 바꿔주는 것**

### "나중에 정형화하면 안 되나?" 논의
```
지금 정형화 (현재 설계):
  + 데이터가 항상 깨끗
  + 실시간 브리핑, 누락 체크 가능
  + 차트 쓸 때 바로 검증 → 정확도 높음
  - 설계가 복잡

나중에 정형화:
  + 당장 단순함
  - 연구할 때마다 전체 차트 재분석 (비용 + 시간)
  - 과거 차트 분석 시 검증 어려움
  - 실시간 기능 불가
```

## AI 역할 정리 문서 작성

### 6가지 AI 역할로 정리 완료 → AI_역할_정리.md

```
1. 데이터 입력: free-text 분석 → 테이블 분배, 신규/기존 구분, 날짜 자동 변환
2. 수정: 검사(원본 수정→전체 반영) vs 진단(이 차트만/전부 선택)
3. 삭제: 방문 삭제 시 검사/영상 보호, 진단은 종결 권장
4. 브리핑/조회: 이력 요약, 누락 체크, 차트 검토 + 동의/반려
5. 연구: SQL로 정형 검색, AI로 free-text 분석, 자연어→SQL 변환
6. 안전 원칙: 할루시네이션 대응, 3중 안전장치, temperature 설정
```

### 역할 분담
```
SQL:  정형 데이터 저장/검색/정렬/통계, 데이터 무결성 보장
AI:   free-text 분석, 정형화, 분배, 비교, 브리핑, 누락 체크
의사: free-text 작성, 의학적 판단, 동의/반려, 테이블 설계, 프롬프트 설계
```

## 추가 설계: 진단 수정/삭제 시 ID 선택 기능

### 시나리오 — 고혈압 의심 vs 고혈압 확진
```
초진: "고혈압"으로 적었지만 약은 안 줌
경과: 혈압 올라서 약 시작
발견: 사실 초진 때는 "고혈압 의심"이었음
→ 약 시작 이전 차트들만 "고혈압 의심"으로 바꾸고 싶음
→ 약 시작 이후 차트의 "고혈압"은 유지하고 싶음
```

### 해결 — 선택지 3가지로 확장
```
진단 수정/삭제 시 AI가 목록을 보여줌:

  □ 진단id=1, 방문id=1 (260110 초진)
  □ 진단id=2, 방문id=2 (260210 경과)
  □ 진단id=3, 방문id=3 (260310 경과)
  ☑ 진단id=4, 방문id=4 (260415 약 시작)

  선택지:
    1. 이 차트만 (1개)
    2. 선택한 차트들 (직접 선택)  ← 새로 추가!
    3. 같은 진단 전부
```

### 적용 결과
```
의사가 2번 선택 → 진단id 1,2,3 체크 → "고혈압 의심"으로 수정:
  진단id=1 (초진)   "고혈압 의심" [종결]
  진단id=2 (경과)   "고혈압 의심" [종결]
  진단id=3 (경과)   "고혈압 의심" [종결]
  진단id=4 (약시작)  "고혈압" [활성]  ← 그대로!
```

### 추가된 함수
```python
진단수정_선택(진단id_리스트, 수정할항목, 새값)   # 선택한 것들만 수정
진단삭제_선택(진단id_리스트)                     # 선택한 것들만 삭제
```

## 핵심 질문과 이해

### 데이터 흐름 관련

**"하나의 기록을 삭제하면 연동된 기록도 다 사라지나?"**
- 입력은 한꺼번에 분배되지만, 저장 후에는 독립적
- 방문 삭제해도 검사결과/영상검사는 유지
- 진단/추적계획도 유지

**"수정하면 연동된 테이블도 수정되나?"**
- 자동이 아니라 AI가 감지 → 의사에게 확인 → 승인 후 반영

**"이전 검사 기록을 다음 차트에 적으면 중복 저장되지 않나?"**
- AI가 기존 테이블과 비교 → 이미 있으면 저장 안 함
- 다르면 의사에게 확인 요청

**"삭제 후 재작성하면 어떻게 되나?"**
- AI가 새 free-text와 기존 테이블 데이터를 비교
- 동일=건너뜀, 충돌=선택지 제시, 새로운 것=확인 후 추가

**"진료 형태가 바뀌면 테이블을 매번 새로 만들어야 하나?"**
- ALTER TABLE로 칸 추가 가능 (Claude Code에게 시키면 됨)

**"일단 free-text로 저장하고 나중에 연구할 때 AI가 정형화하면 안 되나?"**
- 가능하지만, 연구 때마다 전체 재분석 필요 (비용/시간)
- 지금 정형화하면 실시간 기능(브리핑, 누락 체크)도 가능

### 진단 관련

**"진단 종결과 삭제의 차이는?"**
- 종결: 기록 보존, 이력 추적 가능, 의무기록 원칙에 부합
- 삭제: 기록 사라짐, 완전한 오입력일 때만 사용

**"다른 차트와 연동해서 진단을 쓸 수 있으면 좋은데, 수정/삭제까지 연동되면 문제"**
- 맞음. 진단은 검사와 달리 수정/삭제 시 "이 차트만" vs "전부" 선택하게 함

### 진단 수정 심화

**"수정할 진단명이 적용될 id를 선택할 수 있게 하면?"**
- 초진~약 시작 전까지의 "고혈압"을 전부 "고혈압 의심"으로 바꿔야 하는 상황
- 초진 1개만 바꾸는 게 아니라 약 시작 이전 차트 전부 바꿔야 함
- → 진단 목록을 보여주고 의사가 직접 체크하는 방식 추가
- → 진단수정_선택(), 진단삭제_선택() 함수 추가

### 시스템 구조 관련

**"지금 AI가 분배하는 건 아니지?"**
- 맞음. AI 연동은 Week 3. 지금은 테이블과 함수(통로)만 만드는 단계
- 나중에 AI가 붙으면 같은 함수를 호출. 입력 방식만 바뀜.

**"없던 게 새로 들어오면 물어보지 않고 추가하나?"**
- 확인은 받음 ("이거 넣습니다, 맞습니까?")
- 특별한 선택지 없이 목록만 보여주고 확인

## 학습 과정 관찰

### 시스템 설계 사고의 깊이
- "입력은 한꺼번에인데 수정/삭제는 독립적" — 데이터 흐름의 비대칭성 인식
- "검사는 불변이고 진단은 가변" — 데이터 성격에 따른 처리 방식 구분
- "이전 기록을 적으면 중복 저장될까?" — 에지 케이스 사전 예방
- "삭제 후 재작성 시 기존 테이블과 충돌" — 실제 사용 시나리오 시뮬레이션
- "free-text만 지우면 테이블에 고아 데이터가 남는다" — 데이터 정합성 고민

### EMR 실무 경험의 전이
- 의무기록 수정 원칙 (줄 긋고 옆에 새로 적기) → 진단 종결 + 추가 방식
- 검사 결과는 하나의 사실 → 원본 1개 관리 원칙
- 진료 흐름에서의 에지 케이스 → AI 처리 로직으로 변환
