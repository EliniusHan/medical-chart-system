## 코딩 트랙: 데이터베이스 입문

### JSON의 한계 (직접 체험)

**한계 1: 전체를 다 읽어야 함**
- 50명 환자에서 "60세 이상 고혈압"을 찾으려면 50명 전부 메모리에 올려서 하나씩 확인
- 5만 명이면 5만 명 전부 읽어야 함 → 느리고 무거움

**한계 2: 조건 검색마다 함수를 새로 만들어야 함**
- "고혈압 환자 검색" 함수, "60세 이상 검색" 함수, "고혈압+60세 이상" 함수 각각 따로 필요
- 조건 조합이 늘어날 때마다 함수를 계속 추가해야 함

**한계 3: 실시간 조건 변경 불가**
- 프로그램 실행 중에 새로운 조건으로 검색하려면 .py 파일 수정 → 저장 → 재시작 필요
- 다른 사용자는 코드를 수정할 수 없음 (완성된 프로그램을 쓰니까)

### SQLite 데이터베이스의 장점

**SQL은 조건만 바꾸면 됨**
```sql
SELECT * FROM 환자 WHERE 진단 = '고혈압'
SELECT * FROM 환자 WHERE 나이 >= 60
SELECT * FROM 환자 WHERE 진단 = '고혈압' AND 나이 >= 60
-- 새 함수 필요 없이 조건만 자유롭게 조합
```

**프로그램 재시작 없이 실시간 검색**
- SQL 조건만 바꾸면 바로 결과가 나옴
- 나중에 AI가 자연어를 SQL로 변환 → 의사가 아무 조건이나 입력 가능

**Python의 리소스 부담 감소**
- JSON: Python이 전부 다 함 (읽기 + 검색 + 정렬 + 통계)
- SQL: DB 엔진이 무거운 일을 함 → Python은 결과만 받음
- 비유: JSON은 혼자 다 하는 것, SQL은 의무기록실 직원에게 검색을 맡기는 것

**인덱스로 대용량 데이터도 빠르게 검색**
- 책 뒤의 색인(찾아보기)과 같은 원리
- 5만 명이든 50만 명이든 검색 속도가 크게 안 느려짐

### JSON은 데이터베이스가 아님
- JSON = 그냥 파일 (텍스트를 구조화해서 저장)
- SQLite = 진짜 데이터베이스 (테이블, 검색, 정렬, 통계 기능 내장)

## 새로 배운 개념 (코드 리딩으로 학습)

### 경로 관련

**os.path.dirname(__file__)**
- `__file__` = 지금 실행 중인 이 .py 파일 자체
- `os.path.dirname()` = 그 파일이 있는 폴더 경로
- 용도: 터미널 위치와 상관없이 항상 같은 경로를 가리킴
- 아까 겪은 상대경로 문제를 근본적으로 해결

**os.path.join(경로1, 경로2)**
- 경로 조각들을 이어 붙여서 하나의 완전한 경로를 만듦
- `+`로 직접 붙이면 `/`가 빠질 수 있지만, `os.path.join()`은 운영체제에 맞게 자동으로 `/` 추가
- 연결된 파일이 반드시 같은 폴더에 있을 필요는 없음 (하위 폴더도 지정 가능)

### 파일 관련

**with open() as f:**
- `f = open()` + `f.close()`를 합친 안전한 방식
- 들여쓰기 블록이 끝나면 자동으로 파일을 닫아줌
- 닫는 걸 깜빡할 일이 없어서 실무에서 더 많이 사용
- 비유: 차트를 자동 반납하는 시스템

### SQLite 연결 구조

**sqlite3.connect(DB경로)**
- SQLite DB에 연결 (파일 열기와 비슷)
- `conn`은 connection의 줄임말
- 이후 `conn.`으로 시작하는 명령은 전부 이 DB에 대한 명령

**conn.cursor()**
- DB에 명령을 보내고 결과를 받는 도구 (입력창)
- conn = DB에 로그인, cursor = 검색창 열기
- cursor.execute()로 SQL 명령을 실행

**cursor.execute("SQL 명령")**
- SQL 명령을 DB에 보내서 실행
- 진료 비유: 검색창에 명령 입력하고 엔터 치기

**cursor.fetchall() / cursor.fetchone()**
- fetchall() = 검색 결과 전부 가져오기 (여러 명)
- fetchone() = 검색 결과 한 줄만 가져오기 (한 명)
- 가져오기만 함, 출력은 print()가 담당

**conn.commit()**
- 변경사항을 확정 (Git의 commit과 같은 개념)
- commit 전까지는 임시 상태 → 문제 시 전부 취소 가능
- 데이터를 넣거나 수정할 때만 필요, 읽기만 할 때는 불필요
- 비유: 차트에 기록 후 서명 날인

**conn.close()**
- DB 연결 닫기
- 읽기만 했으면 commit 없이 바로 close
- 변경했으면 commit 후 close

### 파일 입출력과 DB의 구조 비교
```
파일:  open() → read()/write() → close()
DB:    connect() → execute()/fetch() → commit() → close()
```

### SQL 기본 문법

**SQL은 자연어가 아닌 정형화된 언어**
- 영어처럼 읽히지만 정해진 문법이 있음
- 선생님은 직접 쓸 줄 몰라도 됨, 읽었을 때 이해하면 충분
- 나중에 AI가 자연어를 SQL로 변환해줌

**핵심 SQL 키워드**
| 키워드 | 의미 |
|--------|------|
| SELECT | 골라줘 (조회) |
| FROM | 여기서 (테이블) |
| WHERE | 조건은 (필터) |
| CREATE TABLE | 테이블 만들어라 |
| DROP TABLE | 테이블 삭제해라 |
| INSERT INTO | 넣어라 |
| GROUP BY | 묶어서 |
| ORDER BY | 정렬해서 |
| COUNT(*) | 개수 세기 |
| AVG() | 평균 |
| ROUND() | 반올림 |
| LIMIT | 결과 개수 제한 |
| DESC | 내림차순 |

**CREATE TABLE — 테이블 구조 만들기**
```sql
CREATE TABLE 환자 (
    id INTEGER PRIMARY KEY AUTOINCREMENT,  -- 자동 번호 (환자 등록번호)
    이름 TEXT,       -- 글자만
    나이 INTEGER,    -- 숫자만
    진단 TEXT,
    수축기 INTEGER,
    이완기 INTEGER
)
```
- TEXT = 글자, INTEGER = 숫자
- PRIMARY KEY = 고유 번호 (중복 불가)
- AUTOINCREMENT = 1, 2, 3... 자동 번호 매김

**DROP TABLE IF EXISTS**
- 테이블이 이미 있으면 삭제 → 에러 방지

**INSERT INTO — 데이터 삽입**
```sql
INSERT INTO 환자 (이름, 나이, 진단, 수축기, 이완기) VALUES (?, ?, ?, ?, ?)
```
- `?`는 보안을 위한 자리표시자 (SQL 인젝션 방지)
- 뒤에 오는 값들이 순서대로 `?` 자리에 들어감

**SELECT — 조건 검색**
```sql
SELECT 이름, 나이 FROM 환자 WHERE 나이 >= 60 AND 진단 = '고혈압'
```
- "환자 테이블에서 / 60세 이상 고혈압인 / 이름과 나이를 / 골라줘"

## 핵심 질문과 이해

### JSON vs DB 관련

**DB도 결국 전체를 다 돌려봐야 하지 않나?**
- Q: "SQL도 5만명을 다 돌려봐야 하는 거 아냐?"
- A: JSON은 파일 전체를 Python 메모리에 올려서 Python이 검색. DB는 DB 엔진이 내부적으로 검색하고 결과만 Python에 전달. Python 입장에서 리소스를 덜 먹음.
- 추가로 DB는 인덱스(색인)가 있어서 전체를 안 돌려봐도 빠르게 찾을 수 있음.

**함수를 AI가 조건에 맞게 변형시키면 DB와 차이가 없지 않나?**
- Q: "함수도 AI가 바꿔주면 되는거 아냐?"
- 스스로 답을 도출: 나만 쓰면 Claude Code로 py 수정 가능하지만, 다른 사람은 완성된 프로그램을 쓰니까 수정 불가. 그리고 py 수정은 저장+재시작 필요하지만 SQL은 실시간 변경 가능.

**Python 리소스 감소**
- Q: "리소스도 덜 먹고 더 빠르겠네?"
- A: 맞음. JSON은 Python이 모든 일을 함. DB는 DB 엔진이 무거운 일을 하고 Python은 결과만 받음.

### 코드 해석 관련

**os.path.join()의 의미**
- Q: "join은 합류시키겠다는 의미야?"
- A: "이어 붙이다"의 의미. 경로 조각들을 이어 붙여서 완전한 경로를 만듦. 자동으로 `/` 추가.

**with open() as f:**
- Q: "f = open()이랑 같은 거 아냐?"
- A: 거의 같지만 with는 자동으로 close()해줌. 닫는 걸 깜빡할 일 없어서 더 안전.

**cursor의 의미**
- Q: "cursor가 검색 준비시키겠다는 의미인가?"
- A: DB에 명령을 보내고 결과를 받는 도구. conn = 로그인, cursor = 검색창 열기.
- "input 명령과 비슷하네" → 비슷하지만 방향이 다름. input은 사람→프로그램, cursor는 프로그램→DB.

**SQL 문법이 자연어인지**
- Q: "저런 명령들은 자연어로 쓴 거야 아니면 정형화된 거야?"
- A: 정형화된 SQL 언어. 영어처럼 읽히게 설계되었지만 정해진 문법이 있음.

**cursor.fetchall()의 역할**
- Q: "출력 준비시키라는 뜻이야?"
- A: "결과를 전부 가져와라". 출력은 print()가 담당. fetchall=전부, fetchone=한 줄만.

**conn.commit()의 역할**
- Q: "commit 명령은 뭐야?"
- A: Git commit과 같음. 변경사항 확정. commit 전까지는 임시 상태. 읽기만 할 때는 불필요.

**conn.close()에 commit이 없는 이유**
- 직접 파악: "읽기만 하는 거니까 저장(commit)할 필요 없이 닫으면 되는 거지?"
- 정확한 이해!

### json_to_sqlite.py와 day4_step3.py의 관계
- Q: "등록하면 JSON에 저장하고 실시간으로 SQLite에도 저장하는 건가?"
- A: 아님. 두 파일은 별개. json_to_sqlite.py는 일회성 변환 도구. 앞으로 해야 할 건 day4_step3.py 자체를 DB 방식으로 바꾸는 것.
