## 새로 배운 개념

### from datetime import datetime
- datetime 도구 상자 안에 datetime이라는 도구가 있음 (이름이 같아서 헷갈림)
- `from datetime import datetime`하면 `datetime.now()` 바로 사용 가능
- `import datetime`만 하면 `datetime.datetime.now()` 두 번 써야 함
- datetime 도구 상자 안의 다른 도구들: date(날짜만), time(시간만), timedelta(기간 계산)
- timedelta: 나중에 AI가 "일주일 후" 같은 표현을 날짜로 변환할 때 활용 가능

### PRAGMA foreign_keys = ON
- SQLite에서 FOREIGN KEY 보호 기능을 켜는 설정 명령
- SQLite는 기본적으로 FOREIGN KEY 체크가 꺼져 있어서 직접 켜줘야 함
- 켜면: 존재하지 않는 환자id로 방문 기록 넣기 → 거부!
- PRAGMA는 SQL 명령이 아니라 SQLite 자체의 설정 명령
- 실제로 있는 데이터끼리만 연결되도록 보호하는 안전장치

### -= 연산자 (줄임말)
```python
나이 -= 1    # 나이 = 나이 - 1 의 줄임말
나이 += 1    # 나이 = 나이 + 1 의 줄임말
```

### return None의 의미
- "계산 실패했다"는 신호를 보내는 것
- print()로 직접 메시지를 출력하지 않고, 호출하는 쪽이 결정하게 함
- 비유: 검사실이 "검체 부적합" 결과지만 보냄 → 환자 설명은 의사가 함

### _ (언더스코어) — 값 안 쓰고 개수만 셀 때
```python
"?" for _ in [1, 2, 3]
# 1, 2, 3 값 자체는 필요 없고
# 그냥 3번 돌려서 "?" 3개 만들려고
# → "?", "?", "?"
```

### .join() — 이어 붙이기
```python
",".join(["?", "?", "?"])     # → "?,?,?"
"-".join(["2026", "03", "28"]) # → "2026-03-28"
```

### IN (?,?,?) — SQL에서 "이 중 하나라도 해당하면"
```sql
WHERE 진단id IN (1, 2, 3)
-- "진단id가 1이거나 2이거나 3인 것"
```
- 선택한 것만 처리할 때 사용
- `?` 개수는 선택한 id 개수에 따라 자동 생성

### placeholder (자리 표시자)
- `?`는 "여기에 값이 들어갈 자리"라는 표시
- SQL 양식(빈 칸)과 실제 값을 따로 보내는 구조
- 보안 목적: DB가 값을 안전하게 처리 (SQL 인젝션 방지)
- 빈 칸 개수를 먼저 알려주고, 그 다음 값을 전달하는 2단계 구조

### [새값] + list(진단id_리스트) — 리스트 합치기
```python
["고혈압 의심"] + [1, 2, 3]
→ ["고혈압 의심", 1, 2, 3]
# ?의 자리에 순서대로 들어갈 값들
```
- list()를 씌우는 이유: 튜플 등 다른 형태일 수도 있어서 확실하게 리스트로 변환

### WHERE 조건 vs IN 선택의 차이
```python
# IN: 내가 고른 것만 (진단id 직접 지정)
WHERE 진단id IN (?, ?, ?)

# WHERE: 조건에 맞는 건 몇 개든 전부 (자동)
WHERE 환자id = ? AND 진단명 = ?
```

## 함수별 역할과 print() 원칙

### print() 사용 원칙
```
input()이 있는 함수 (사용자와 직접 소통)  → print() OK
  예: 숫자입력()

input()이 없는 함수 (계산/DB 작업만)      → return만 (print 안 함)
  예: 나이계산(), 환자등록(), 진단수정()
```

### 함수 계층 구조
```
사용자 ↔ day4_step3.py ↔ 숫자입력()           (직접 소통)
사용자 ↔ day4_step3.py ↔ 환자등록() ↔ 나이계산() (간접)
```
- 숫자입력()은 사용자와 바로 대화하는 함수라 print() 자연스러움
- 나이계산()은 안쪽에 있어서 결과만 돌려주고 밖에서 표시

### 일관성 문제 발견! (코드 리뷰 성과)
```
나이계산()     → return None (원칙대로) ✅
숫자입력()     → print() (사용자와 직접 소통하니까 OK) ✅
진단수정_단일() → print() + return False (섞여있음) ⚠️
```
- utils.py의 수정/삭제 함수들에서 허용항목 체크 시 print()를 직접 함
- 원칙대로라면 return False만 하고 day4_step3.py에서 print() 해야 함
- 지금 터미널에서는 문제 안 되지만, 나중에 웹앱 전환 시 문제됨
- → Claude Code에 수정 프롬프트 전달 완료

### 터미널 vs 웹앱에서의 차이
```
터미널:  utils.py에서 print() → 터미널에 나옴 (사용자 보임)
웹앱:    utils.py에서 print() → 서버 로그에만 나옴 (사용자 안 보임!)
```

## 핵심 질문과 이해

### datetime 관련

**"왜 from datetime import datetime이지? 그냥 import datetime 하면 안 돼?"**
- datetime 도구 상자와 안의 도구 이름이 같아서
- from 없이 import만 하면 datetime.datetime.now()로 두 번 써야 함

### PRAGMA 관련

**"PRAGMA가 뭐야?"**
- SQLite 자체 설정 명령. "실제 있는 환자끼리만 연결해라" 보호 기능 켜기

### return None vs print() 관련

**"나이계산에서 왜 None을 반환해? print 하면 되지 않아?"**
- utils.py는 계산만, 출력은 day4_step3.py가 담당 (역할 분리)
- 나중에 웹앱으로 바꾸면 print()가 의미 없어짐

**"진단수정에서는 왜 print를 직접 하지?"**
- Claude Code가 만든 코드의 일관성 문제 → 수정 요청함
- AI가 만든 코드도 완벽하지 않다는 걸 직접 발견한 코드 리뷰 성과!

### placeholder 관련

**"뭘 3번 돌리라는 거야?"**
- 진단id 3개 선택하면 ? 3개 필요 → 자동으로 ?,?,? 생성

**"?는 뭔데?"**
- "여기에 값이 들어갈 자리" 표시. 보안을 위해 양식과 값을 따로 보내는 구조.

**"list에서도 ?로 보내야 하는 거 아냐?"**
- ?는 SQL 양식의 빈 칸. list는 그 빈 칸에 채울 실제 값. 둘이 한 쌍이어야 작동.
- 1단계: 빈 칸 개수 알려줌 (?,?,?) → 2단계: 채울 값 전달 (1,2,3)

**"WHERE 조건이랑 IN이랑 뭐가 달라?"**
- IN: 내가 직접 고른 id만 (진단수정_선택)
- WHERE: 조건에 맞는 건 자동으로 전부 (진단수정_전체)

### 출력 관련

**"전체 기록을 다 return 받으면 어떻게 출력해?"**
- 큰 딕셔너리 안에 각 테이블 데이터가 리스트로 들어있음
- day4_step3.py에서 항목별로 for문 돌면서 출력

**"원래 차트에 쓴대로 합쳐서 볼 수는 없어?"**
- 가능! free_text에 원본이 보존되어 있으니 방문일 순서대로 출력하면 됨
- 나중에 프론트엔드에서 차트 원본/정리 보기 탭 전환 가능
