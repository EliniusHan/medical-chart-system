## 새로 배운 개념

### 함수 기본값 (Default Parameter)
```python
def 환자선택(안내문="환자 이름: "):
#                   ↑ 기본값: 안 주면 이걸 사용
```
- 호출할 때 안내문을 안 주면 → "환자 이름: " 자동 적용
- 안내문을 주면 → 그걸 사용
- 같은 함수를 여러 곳에서 다른 메시지로 재사용 가능:
```python
환자선택()                      # → "환자 이름: "
환자선택("수정할 환자: ")        # → "수정할 환자: "
환자선택("삭제할 환자: ")        # → "삭제할 환자: "
```

### .strip() — 앞뒤 공백 제거
```python
"  김철수  ".strip()    # → "김철수"
```
- 사용자가 실수로 공백 넣어도 검색 가능하게 해줌

### 문자열 반복 (*)
```python
"=" * 50   # → "=================================================="
```
- 구분선 만들 때 사용

### v (변수 이름 약자)
```python
for v in 기록["방문"]:    # v = visit (방문)의 약자
    v['수축기']           # 이 방문의 수축기
```
- Claude Code가 영어 약자로 쓴 것, `방문`이라고 써도 됨

### 선택 = 숫자입력() 구조
```python
선택 = 숫자입력(" 환자 번호 선택: ", 1, len(결과))
return 결과[선택 - 1]    # -1은 리스트가 0번부터 시작하니까
```
- 동명이인 있을 때 번호로 선택하는 구조
- 숫자입력() 함수 안에 input(), print()가 이미 들어있음
- 호출하는 쪽에서는 결과만 받으면 됨

## 검사결과 개별 id 이해

### 같은 날 검사도 항목별로 따로 id 부여
```
저장 (DB 내부):
  검사id=1, 260310, FBS, 108
  검사id=2, 260310, A1c, 5.8
  검사id=3, 260310, LDL, 162

출력 (사용자가 보는 화면):
  260310  FBS 108, A1c 5.8, LDL 162
```

### 이유: 연구 시 항목별 추적 필요
```sql
-- 항목별 저장이면 SQL 한 줄로 가능
SELECT 검사시행일, 결과값 FROM 검사결과
WHERE 검사항목='A1c' ORDER BY 검사시행일

-- 통으로 저장이면 매번 AI가 텍스트 분석해야 함
```

### 수정/삭제 시 흐름
```
수정: AI가 항목별로 비교 → 바뀐 것만 확인 (동일한 건 건너뜀)
삭제: 같은 날짜 항목 목록 보여주고 → 전부/선택 선택지
```

## 결과값 TEXT vs 숫자 통계 문제

### 문제 발견
- 결과값이 TEXT로 저장 ("124", "C1 negative" 등 혼재)
- SQL에서 바로 AVG() 등 통계 함수 사용 불가

### 해결 방안 (Week 4에서 적용 예정)
```
검사결과 테이블에 '결과수치' 칸 추가:
  FBS:  결과값="124",         결과수치=124
  BMD:  결과값="L -1.1",      결과수치=-1.1
  유방: 결과값="C1 negative",  결과수치=NULL
```
- 숫자인 항목은 결과수치(REAL)에도 저장 → SQL 통계 가능
- AI가 입력 시 숫자 여부 판단 → 결과수치에 자동 저장

### 당장은 대안으로 처리 가능
```sql
-- SQL에서 TEXT→숫자 변환
SELECT AVG(CAST(결과값 AS REAL)) FROM 검사결과 WHERE 검사항목='FBS'
-- 또는 AI가 분석 시 알아서 숫자로 처리
```

## 진단 설계 심화

### 진단 상태 변경: 4개 → 3개
```
기존: 활성, 관해, 종결
변경: 활성, 의심, 종결

이유: "관해"가 모호함
  - 고혈압이 좋아져서 치료 안 하는 건지?
  - 약 먹으면서 잘 조절되는 건지?
  → 의사 입장에서 헷갈림
  → 세부 내용은 진단명 자체에 포함시키면 됨
    예: "#고혈압", "#고혈압(LSM)", "#고혈압(치료 안 함)"
```

### 진단 상태 3가지 정의
```
활성   → 현재 해당되는 확진 진단 (#고혈압, #고혈압(LSM))
의심   → 아직 확진 안 됨 (#r/o 고혈압, #고혈압 의심)
종결   → 더 이상 해당 안 됨 (오진, 완치 등)
```

### 진단명 + 비고 분리 저장
```
의사가 쓰는 방식 (자유):
  #고혈압(LSM)
  #고혈압 - LSM
  #r/o 고혈압

AI가 분배할 때:
  "#고혈압(LSM)"  → 진단명="고혈압", 비고="LSM", 상태="활성"
  "#r/o 고혈압"   → 진단명="고혈압", 비고="", 상태="의심"

장점: 통계 돌릴 때 진단명으로 묶을 수 있음
  SELECT COUNT(*) FROM 진단 WHERE 진단명='고혈압'
  → 고혈압 전체 (LSM이든 약이든 전부 포함)
```

### 다중 사용자 문제 — 의사마다 표현이 다름
```
A 의사: "#고혈압(LSM)"
B 의사: "#HTN - lifestyle modification"
C 의사: "#essential hypertension"
→ AI가 표준 진단명으로 매핑해야 함
```

### 표준코드(KCD/ICD) 도입
```
진단 테이블에 표준코드 칸 추가:
  진단명="고혈압", 표준코드="I10"     ← 매핑 가능
  진단명="경동맥 비후", 표준코드=NULL  ← 매핑 불가

통계 시 2단계:
  1단계: 코드 있는 진단 → SQL로 자동 포함
  2단계: 코드 NULL → AI가 후보 추려서 의사에게 제시 → 포함/제외 판단

NULL은 빠지는 게 아니라 "따로 모아서 확인받는 것"
```

## 핵심 질문과 이해

### 코드 리딩 관련

**"환자선택에서 안내문 변수가 정의된 적 없는데?"**
- 함수의 매개변수는 함수 정의할 때 거기서 만들어짐
- 기본값이 있으면 호출 시 안 줘도 됨

**"숫자입력 호출하는데 왜 print나 input이 없지?"**
- 숫자입력() 함수 안에 이미 다 들어있음
- 호출하는 쪽에서는 결과만 받으면 됨

**"v는 뭐지?"**
- visit(방문)의 약자. Claude Code가 영어 약자로 쓴 것.

### 검사결과 관련

**"검사 id가 같은 날인데 항목별로 따로 저장되네?"**
- 연구 시 항목별 추적이 가능해야 하므로
- 출력할 때는 같은 날짜끼리 묶어서 보여줌

**"수정할 때는 free-text로 쓰면 각 id가 알아서 수정되나?"**
- AI가 항목별로 비교 → 바뀐 것만 확인 → 동일한 건 건너뜀

**"삭제할 때는 항목별로 다 물어보나?"**
- 같은 날짜 항목 목록 보여주고 → 전부/선택 선택지

**"결과값이 TEXT면 통계 못 돌리지 않나?"**
- Week 4에서 결과수치(REAL) 칸 추가 예정
- 당장은 CAST() 변환이나 AI 분석으로 대체 가능

### 진단 설계 관련

**"활성/관해/종결에서 '의심'을 추가해야 하지 않나?"**
- 맞음. 의심과 확진은 의학적으로 완전히 다름

**"'관해'가 필요한가?"**
- 모호함 → 제거. 세부 내용은 진단명 자체에 포함 (#고혈압(LSM))

**"비고사항은 무조건 ()안에 쓰라고 해야 하나?"**
- 아니요. AI가 알아서 파싱 (괄호, 쉼표, 대시 등 인식)
- 의사 습관 안 바꿔도 됨

**"다중 사용자면 의사마다 표현이 달라서 문제"**
- 표준코드(KCD/ICD) 매핑으로 해결
- AI가 자동 매핑, 안 되면 NULL

**"NULL이면 통계에서 빠지지 않나?"**
- 빠지는 게 아니라 따로 모아서 AI가 후보 추림 → 의사 판단

### 기능 확장 관련

**"외부 기록 사진 분석 기능 가능한가?"**
- 가능. Claude API가 이미지 읽을 수 있음.
- 기존 코드 수정 없이 새 모듈만 추가 (import로 연결)
