## 코딩 트랙: 에러 처리

### 배운 개념

**try/except**
- "이 코드를 시도(try)해보고, 에러가 나면 죽지 말고 이걸(except) 대신 실행해라"
- 진료 비유: "이 약을 투여해봐(try). 부작용 나면(except) 즉시 이 약으로 교체해"
```python
try:
    나이 = int(input("나이: "))
except ValueError:
    print("⚠ 숫자를 입력해주세요!")
```

**try/except + while 조합**
- try/except만으로는 에러를 잡아줄 뿐, 다시 입력받으러 돌아가지 않음
- while True와 합쳐서 "제대로 입력할 때까지 반복"하는 구조
```python
while True:
    try:
        나이 = int(input("나이: "))
        break                        # 성공하면 탈출
    except:
        print("⚠ 숫자를 입력해주세요!")  # 실패하면 다시 while로
```

**에러 종류별 대응**
- `except:` → 모든 에러를 잡음 (편하지만 진짜 문제도 숨길 수 있음)
- `except ValueError:` → ValueError만 잡음 (더 정확한 대응)
- 진료 비유: "모든 통증에 진통제"가 아니라 "두통이면 이 약, 복통이면 저 약"

**에러 메시지 읽는 법**
- 항상 맨 아래 줄부터 읽기 (거기에 원인이 있음)
- 검사 결과지의 결론부터 읽는 것과 같음
```
Traceback (most recent call last):
  File "day4_step3.py", line 52, in <module>     ← 위치 (52번째 줄)
    나이 = int(input("나이: "))                    ← 어떤 코드
ValueError: invalid literal for int() with base 10: 'abc'  ← 원인 (핵심!)
```

**주요 에러 유형**
| 에러 | 의미 | 진료 비유 |
|------|------|----------|
| SyntaxError | 코드 문법 오류 (따옴표, 들여쓰기) | 처방전 양식 오류 |
| ValueError | 잘못된 값 (숫자 자리에 문자) | 잘못된 값 투여 |
| NameError | 정의 안 된 변수 사용 | 존재하지 않는 약 처방 |
| KeyError | 딕셔너리에 없는 키 조회 | 차트에 없는 칸 조회 |
| FileNotFoundError | 없는 파일 열기 시도 | 없는 차트 꺼내려 함 |
| TypeError | 타입이 안 맞음 | 경구약을 주사기에 넣으려 함 |
| ZeroDivisionError | 0으로 나누기 | 불가능한 계산 |
| json.JSONDecodeError | 깨진 JSON 파일 읽기 | 훼손된 차트 읽기 |

**에러 감지는 Python이 알아서 함**
- int()는 숫자가 뭔지 알아서 "abc"를 넣으면 스스로 ValueError를 냄
- json.load()는 JSON 형식을 알아서 깨진 파일이면 스스로 JSONDecodeError를 냄
- 우리가 할 일은 "에러가 나면 어떻게 대처할지"만 정해주는 것

### 직접 작성한 코드

**숫자입력() 함수 — 첫 번째 시도**
```python
def 숫자입력(질문):
    while True:
        질문 = {나이, 수축기, 이완기}
        if 질문 == "나이":
            try:
                나이 = int(input("나이: "))
                break
            except:
                print("숫자를 입력하세요!")
        elif 질문 == "수축기":
            ...
```
→ 각 경우를 나눠서 처리하려는 발상. 작동은 하지만, 함수의 재료(매개변수)를 활용하면 훨씬 간결해짐.

**숫자입력() 함수 — 개선된 버전 (범위 검증 포함)**
```python
def 숫자입력(질문, 최소, 최대):
    while True:
        try:
            숫자 = int(input(질문))
            if 최소 <= 숫자 <= 최대:
                return 숫자
            else:
                print(f"⚠ {최소}~{최대} 사이의 숫자를 입력해주세요.")
        except ValueError:
            print("⚠ 숫자를 입력해주세요!")
```
→ 함수 하나로 나이, 수축기, 이완기 등 모든 숫자 입력을 처리. 호출할 때 범위만 다르게 지정.

**범위 검증 — 함수 안에서 분기하는 방식도 시도**
```python
def 숫자입력(질문):
    while True:
        try:
            숫자 = int(input(질문))
            if 질문 == "나이: " and 0 <= 숫자 <= 150:
                return 숫자
            elif 질문 == "수축기 혈압: " and 숫자 > 0 or 질문 == "이완기 혈압: " and 숫자 > 0:
                return 숫자
            else:
                print("0-150 사이의 숫자를 입력하세요")
        except ValueError:
            print("숫자를 입력하세요")
```
→ 작동은 하지만 한계: 현재 정의된 변수에만 작동, 새 항목 추가 시 함수를 매번 수정해야 함.
→ 재료(매개변수) 방식은 함수 수정 없이 호출만 추가하면 확장 가능.

**혈압 관계 검증 (직접 작성, 수정 없이 완벽)**
```python
elif 명령 == "등록":
    이름 = input(" 환자 이름: ")
    나이 = 숫자입력("나이: ", 0, 130)
    진단 = input("진단명: ")
    while True:
        수축기 = 숫자입력("수축기 혈압: ", 0, 250)
        이완기 = 숫자입력("이완기 혈압: ", 0, 180)
        if 수축기 > 이완기:
            break
        else:
            print("이완기 혈압이 수축기 혈압보다 클 수 없습니다. 다시 입력하세요.")

    환자 = {
        "이름": 이름,
        "나이": 나이,
        "진단": 진단,
        "수축기": 수축기,
        "이완기": 이완기
    }
    환자목록.append(환자)
    저장하기(환자목록)
    print(f" {이름} 환자 등록 완료! (총 {len(환자목록)}명)")
    print("")
```
→ 3단계 검증 구현: 숫자인가(try/except) → 범위 안인가(함수 내 if) → 두 값의 관계가 맞나(등록 코드 내 while)

**불러오기() 함수 에러 처리 (직접 설계)**
```python
def 불러오기():
    try:
        if os.path.exists("환자DB.json"):
            파일 = open("환자DB.json", "r")
            환자목록 = json.load(파일)
            파일.close()
            print(f"-> 기존 데이터 {len(환자목록)}명 불러옴!")
            return 환자목록
        else:
            print("-> 기존 데이터 없음. 새로 시작합니다.")
            return []
    except:
        print("데이터 파일이 손상되었습니다. 새로 시작합니다.")
        return []
```
→ 제시된 버전과 순서만 다르고 결과는 동일. try가 if와 json.load()를 한꺼번에 감싸서 코드가 간결.

### 핵심 질문과 이해

**try/except만으로는 부족한 이유**
- Q: "숫자를 입력하세요가 뜬 다음에 다시 처음으로 돌아가야 하는데, 그냥 종료됐어"
- A: try/except는 에러를 잡아줄 뿐, 다시 돌아가는 기능은 없음. while과 합쳐야 반복 가능.

**숫자입력() 함수의 설계 — 개별 분기 vs 재료(매개변수) 방식**
- Q: "항목마다 최소, 최대값이 다른데 개별화를 못하지 않나?"
- A: 호출할 때마다 다른 값을 넣으니까 개별화가 됨. `숫자입력("나이: ", 0, 150)`, `숫자입력("수축기: ", 1, 300)` 등.

**함수 안에서 분기하는 방식의 한계**
- Q: "질문 텍스트로 나이/혈압을 구분하면 안 되나?"
- A: 작동은 하지만, 새 항목 추가 시 함수를 매번 수정해야 함. 재료 방식은 함수 수정 없이 호출만 추가하면 확장 가능. 현재 정의된 변수에만 한정됨.

**이완기 > 수축기 방지**
- Q: "이완기 혈압이 수축기보다 크면 안 되는데, 이 오류는 어떻게 방지할까?"
- A: 숫자입력() 함수 안에서는 해결 불가 — 수축기 입력 시점에 아직 이완기 값이 없으므로. 두 값 모두 입력된 후 실행 코드에서 while로 관계를 검증.
- Q: "이건 함수에 넣으면 안 되고 실행 명령에 넣어야겠네?"
- A: 맞음. 함수는 개별 값 검증, 실행 코드는 관계 검증. 역할 분리.

**JSON 파일 손상 감지**
- Q: "DB.json이 깨져있다는 걸 Python으로 어떻게 표현해서 알려주지?"
- A: Python에 알려줄 필요 없음. json.load()가 JSON 형식을 이미 알고 있어서 깨진 파일이면 스스로 JSONDecodeError를 냄. 우리는 대처법만 정해주면 됨.

**불러오기 함수 구조 — 두 가지 접근법**
- 제시된 버전: 파일 찾기 → 있으면 읽기(try) → 깨졌으면(except) 새로 시작 → 없으면 새로 시작
- 선생님 버전: 전체를 try로 감싸기 → 파일 찾기 → 있으면 읽기 → 없으면 새로 시작 → 뭐든 에러나면(except) 새로 시작
- 둘 다 정상 작동. 선생님 버전이 오히려 코드가 간결한 장점 있음.

**or와 and 우선순위**
```python
# 잘못된 경우 (첫 번째 시도)
질문 == "수축기" or 질문 == "이완기" and 숫자 > 0
# → (수축기) or (이완기 and 숫자>0) → 수축기면 숫자 체크 없이 통과

# 올바른 경우 (두 번째 시도)
질문 == "수축기" and 숫자 > 0 or 질문 == "이완기" and 숫자 > 0
# → (수축기 and 숫자>0) or (이완기 and 숫자>0) → 의도대로 작동
```

**코드를 보고 베끼면서 하는 게 맞는 건가**
- Q: "혼자 하는 게 아니라 이전 코드 보면서 베껴가면서 하는 거야..."
- A: 실제 개발자들도 이전 코드 참고, Stack Overflow 복사, AI 활용이 일상. 중요한 건 베끼면서도 이해하고 있다는 것. "이완기가 수축기보다 크면 안 되니까 while로 감싸야겠다"는 판단 — 그건 베끼는 게 아님.

## AI 트랙: 컨텍스트 윈도우, temperature, max_tokens

### 컨텍스트 윈도우 (Context Window)
- AI가 한 번에 읽을 수 있는 최대 토큰 수 = "작업 기억력의 한계"
- Claude: 약 200,000 토큰, Gemini: 약 1,000,000~2,000,000, GPT-4: 약 128,000
- 프로젝트 적용: 시스템 프롬프트 + 환자 기록 + free-text + AI 응답이 모두 이 안에 들어가야 함
- 환자 차트 1건 분석은 약 5,000~13,000 토큰으로 충분히 여유 있음
- DB에서 필요한 것만 꺼내서 보내는 것이 중요 → 지금 배우는 DB 설계와 직결

### temperature
- AI 응답의 창의성/일관성 조절 다이얼
- 0.0 = 매번 거의 같은 답 (보수적, 정확)
- 1.0 = 매번 다른 답 (창의적, 가끔 엉뚱)
- 프로젝트 적용:
  - 차트 검토: 낮게 (0.0~0.2) — 일관된 기준으로 검토
  - 완성본 차트: 낮게 (0.1~0.3) — 의사 스타일대로 일관되게
  - 연구 데이터 해석: 중간 (0.3~0.5) — 다양한 시각 가능
- 의료 AI에서는 대부분 낮은 temperature가 적합

### max_tokens
- AI 응답의 최대 길이 제한
- 크게 설정해도 실제 생성한 만큼만 과금 (비용 걱정 불필요)
- 너무 작게 설정하면 답변이 중간에 잘림
- 프로젝트 적용: 브리핑 500, 차트 검토 2000, 완성본 차트 4000 정도

### Google AI Studio 실험 경험
- temperature 변경 시 의료 프롬프트 응답이 "조금 달라지는" 것 확인함
- 의료에서는 일관성이 중요하므로 낮은 temperature 권장
