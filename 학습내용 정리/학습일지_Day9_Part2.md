## 테이블 설계 (의사 관점의 설계)

### 선생님 차트 4개 분석 후 구조 파악

**차트에서 발견된 일관된 구조:**
```
1. 상단: 진단 목록 (#으로 표시)
2. 검사 수치: 날짜 + 항목별 수치
3. 활력징후: BP, 키/몸무게/BMI
4. 생활습관: 음주/흡연/운동/식이
5. 본문: free-text 진료 기록
6. 하단: 처방/계획
   * = 처방
   ** = 추적 예정 검사
   *** = 약 처방 기간
```

### 최종 확정 테이블 구조 (6개)

**테이블 1: 환자 (기본 정보 — 잘 안 바뀌는 것)**
```
환자id (자동번호)
이름, 생년월일(YYYYMMDD), 성별, 가족력, 약부작용이력
```

**테이블 2: 진단 (문제 목록)**
```
진단id (자동번호)
환자id, 방문id
진단명, 상태(활성/관해/종결), 비고
```

**테이블 3: 방문 (매 방문 기록)**
```
방문id (자동번호)
환자id, 방문일(YYMMDD)
수축기, 이완기, 심박수, 키, 몸무게, BMI
흡연, 음주, 운동
free_text, 처방요약
```

**테이블 4: 검사결과 (혈액검사용)**
```
검사id (자동번호)
환자id, 검사시행일(YYMMDD)
검사항목, 결과값, 단위, 참고범위
```

**테이블 5: 영상검사 (영상/특수검사용)**
```
영상id (자동번호)
환자id, 검사시행일(YYMMDD)
검사종류, 결과요약, 주요수치
```

**테이블 6: 추적계획**
```
추적id (자동번호)
환자id, 방문id, 예정일(YYMMDD)
내용, 완료여부(0=미완료, 1=완료)
```

## 설계 과정에서의 핵심 논의

### 키/체중/흡연/음주/운동의 위치
- 초기: 환자 테이블에 배치
- 선생님 피드백: "내원할 때마다 확인하는 항목"
- 수정: 방문 테이블로 이동

### 나이 vs 생년월일
- 문제: Claude Code가 나이 53을 "7300"(1973년)으로 저장
- 해결: 생년월일을 YYYYMMDD로 저장 → 나이는 자동 계산
- 장점: 매년 업데이트 불필요, 특정 시점의 나이도 정확히 계산 가능

### 날짜 형식 YYMM vs YYMMDD
- 선생님은 보통 YYMM으로 기록
- 하지만 같은 달에 여러 번 방문하는 경우가 많음
- 결론: YYMMDD로 통일 (AI가 자동 변환 가능)

### 검사결과 테이블 — 오더일/검사시행일/확인일 논의
- 초기안: 오더일 + 검사시행일 + 확인일 전부 저장
- 선생님 판단: "검사시행일만 있으면 충분"
  - 오더는 추적계획 테이블에
  - 확인은 방문 free_text에
  - 각 테이블이 자기 역할만 하면 됨
- 최종: 검사시행일만 저장

### 검사사유 칸 추가 여부
- 제안: 검사결과에 "검사사유" 칸 추가
- 선생님 판단: "불필요. 의사는 왜 검사하는지 차트에 다 적으니까"
- AI가 필요할 때 free_text에서 추출하면 됨

### 방문id와 검사id의 관계
- 선생님 질문: "방문id를 최종 결과 보는 날에만 연결해도 되나?"
- 선생님 판단: "모든 방문에 방문id 부여하고, 검사는 날짜로 별도 관리"
- 결론: 방문 테이블과 검사결과 테이블은 독립적으로 운용

### 영상검사 테이블 추가
- 혈액검사와 별도로 영상/특수검사 테이블 필요
- 혈액검사: 항목별로 행 1줄 (값이 숫자 하나)
- 영상검사: 검사종류별로 행 1줄 (값이 복잡한 서술)
- 구조: 검사종류(칸) + 결과요약(free-text) + 주요수치(선택적)
- 이유: SQL로 검사종류별 필터링 + AI가 결과요약 분석

### 혈액검사 vs 영상검사 저장 방식 차이
```
혈액검사: 항목별 행 → 숫자값 → SQL로 직접 비교/통계 가능
영상검사: 종류별 행 → free-text → AI가 분석해서 추출
```

## DB 구조와 AI의 역할 분담

### 정형 데이터 vs 비정형 데이터
```
정형 (SQL 칸): 이름, 나이, BP, 검사수치 → SQL로 바로 검색
비정형 (free-text): 진료 기록 본문 → AI가 분석해서 추출
```

### 데이터 입력 흐름 (향후 시스템)
```
의사: free-text로 편하게 씀 (습관 안 바뀜)
AI: 분석해서 정형화 + 테이블에 분배
의사: "맞아?" 확인 → 동의하면 저장
```

### AI의 날짜 자동 변환
```
의사: "일주일 후에 와서 검사합시다"
AI: 오늘 260115 → 일주일 후 260122로 변환
AI: "260122로 추적계획에 넣을까요?" → 의사 확인 후 저장
```

## 새로 배운 개념

### FOREIGN KEY
- 다른 테이블의 번호를 참조하는 것
- PRIMARY KEY = 이 테이블의 주인 번호 (원본)
- FOREIGN KEY = 다른 테이블의 번호를 빌려온 것 (참조)
- DB가 자동으로 관계 보호: 존재하지 않는 환자id는 넣을 수 없음

### 하나의 DB 파일 안에 여러 테이블
- 환자DB.db 파일 1개 안에 6개 테이블 전부 저장
- 같은 DB 안에 있어야 JOIN으로 한 번에 연결 가능
- 별도 DB면 각각 열어서 수동 매칭 필요 → JSON 때와 같은 문제

### JOIN의 실전 활용
- 여러 테이블을 연결해서 한 번에 조회
- ON 절로 "어떤 칸으로 매칭할지" 지정
```sql
SELECT 환자.이름, 방문.방문일, 검사결과.검사항목, 검사결과.결과값
FROM 환자
JOIN 방문 ON 환자.환자id = 방문.환자id
JOIN 검사결과 ON 방문.방문id = 검사결과.방문id
```

### ALTER TABLE
- 기존 테이블에 새 칸 추가 가능 (진료 형태 변경 시)
- Claude Code에게 시키면 됨
- 기존 데이터는 유지, 새 칸은 비어있음(NULL)

### .db 파일의 성격
- 바이너리 파일이라 VS Code 텍스트 편집기로 열 수 없음
- SQL로만 접근 가능
- Claude Code에게 "테이블 구조 보여줘"라고 확인

### PR (Pull Request)
- "이 변경사항을 메인 코드에 반영해줘"라고 요청하는 것
- 팀 작업용 → 혼자 작업하는 지금은 불필요
- commit + push만 하면 충분

## 학습 과정 관찰

### 의사만 할 수 있는 설계 판단들
- 키/체중/흡연/음주/운동 → "내원마다 확인하는 항목" → 방문 테이블로 이동
- 검사시행일만 저장 → "오더는 계획에, 확인은 free_text에" → 각 테이블 역할 분리
- 검사사유 칸 불필요 → "의사는 다 적으니까 AI가 free_text에서 추출"
- 영상검사 별도 테이블 필요 → "결과가 길어서 차트에 같이 넣기 부담"
- 방문id와 검사id 독립 운용 → 실제 진료 패턴 반영 (당일검사, 예약검사 등)

### 스스로 문제를 제기하고 해결
- "검사결과를 연구용으로 쓸 때 목적 매칭이 될까?" → "free_text에 있으니 가능"
- "날짜를 안 쓰고 '일주일 후'라고만 쓰면?" → "AI가 변환해주면 됨"
- "진료 형태가 바뀌면 테이블을 새로 만들어야 하나?" → "ALTER TABLE로 칸 추가"
- "방문id를 결과 확인 날에만 연결해도 되나?" → "아니, 모든 방문에 부여하고 검사는 독립"
- "오더일/시행일/확인일 다 필요한가?" → "시행일만 있으면 충분, 나머지는 다른 테이블에"
