## 코딩 트랙 + 에이전트 트랙: Git & GitHub

### 배운 개념

**Git — 코드의 수정 이력 추적 시스템**
- 커밋(commit)할 때마다 현재 상태의 스냅샷이 내 컴퓨터에 저장됨
- 이전 커밋 시점으로 되돌리기 가능 → "코드의 타임머신"
- 진료 비유: 차트에 서명하고 수정 이력 남기기

**GitHub — Git의 클라우드 백업**
- Git은 내 컴퓨터에서만 이력 관리
- GitHub는 그 이력을 인터넷에 백업하는 곳
- 컴퓨터 고장 나도 GitHub에서 복구 가능
- 진료 비유: 차트 사본을 외부 보관소에 보내는 것

**Git vs GitHub 관계**
```
Git    = 내 컴퓨터에서 타임머신 (이력 관리, 되돌리기)
GitHub = 그 타임머신의 클라우드 백업 (안전 보관)
```

**커밋 메시지 (Commit Message)**
- 커밋 = 현재 상태를 저장 (차트에 서명 날인)
- 커밋 메시지 = 이번에 뭘 했는지 적는 한 줄 메모
- 예시: "환자 관리 시스템 초기 버전", "통계 기능 추가", "버그 수정"
- 나중에 이력을 볼 때 "이때 이런 작업을 했구나" 파악 가능

**Git 기본 명령어**
```
git init     → 초기화 (Git 시작)
git add      → 추가 (변경 파일 준비)
git commit   → 확정 (현재 상태 저장)
git push     → 밀어올리기 (GitHub에 업로드)
```

**.gitignore — "이 파일들은 Git에서 무시해라" 목록**
- Mac 시스템 파일(.DS_Store) 등 쓸데없는 파일 제외
- API 키(.env), 환자 데이터(*.db) 등 민감한 파일 제외
- 진료 비유: 차트 보관실에 "이건 보관하지 마" 목록 붙여두기

### Claude Code로 Git 실행
- "이 폴더를 Git으로 관리하기 시작해줘" → git init + add + commit 자동 실행
- ".gitignore 설정해줘" → 불필요한 파일 자동 제외
- "GitHub에 새 저장소 만들어서 올려줘" → repository 생성 + push 자동
- 매일 작업 끝에 "변경사항 커밋하고 GitHub에 push해줘" 한 줄이면 백업 완료

### 핵심 질문과 이해

**Git과 GitHub의 관계**
- Q: "방금 한 건 git으로 py들을 올린 거 아니야? 아니면 설정만 해둔 상태?"
- A: 내 컴퓨터 안에서만 저장한 것. 어디에도 올리지 않았음. GitHub에 올려야 클라우드 백업이 됨.

**Claude Code로 버전 복원 가능 여부**
- Q: "Claude에게 이전 버전을 불러와서 교체해줘라고 하면 GitHub에서 내려받는 건가?"
- A: 아님. 내 컴퓨터의 Git 이력에서 가져옴. GitHub는 비상시 백업용. 평소에는 로컬 Git에서 이력 관리.

**커밋 메시지의 의미**
- Q: "커밋 메시지는 무슨 뜻이지?"
- A: commit message의 한글 표현. "이번에 뭘 했는지 적는 메모". Git 용어는 전부 영어.

**.gitignore의 역할**
- Q: "Mac에서 자동으로 만들어내는 쓸데없는 코드들을 git에 안 들어오게 목록을 지정해두는 건가?"
- A: 맞음. 추가로 쓸데없는 파일뿐 아니라 올리면 안 되는 민감한 파일(API 키, 환자 데이터)도 제외 가능.

## 환경 설정 이슈

### 터미널 경로
- py 파일들을 ~/Desktop/Python 폴더에서 관리하기로 결정
- 앞으로 모든 파일은 이 폴더에 저장

### 무선 키보드 ~ 입력 문제
- Windows용 무선 키보드를 Mac에 연결하여 사용 중
- `~` 입력 시 `±`가 출력되는 문제 발생
- 원인: Windows 키보드 레이아웃과 Mac 키 배열 차이
- 해결: 시스템 설정 → 키보드 → 키보드 종류 변경에서 재설정
- 대안: `~` 대신 전체 경로 `/Users/deokjaehan/Desktop/Python` 사용 가능

### 터미널 기본 명령어
- `cd` = change directory (폴더 이동)
- `ls` = list (파일 목록 보기, DOS의 DIR과 동일)
- `&&` = 앞 명령 성공하면 다음 명령도 실행

## 에이전트 트랙: Claude.ai / Claude Code / Cowork 비교

### 세 도구의 차이점

**Claude.ai (브라우저 채팅)**
- 브라우저, 폰에서 접속 (어디서든)
- 질문하면 텍스트로 답변
- 내 컴퓨터 파일에 접근 못함 (업로드해야 함)
- 비유: 전화로 전문가에게 상담

**Claude Code (터미널)**
- 터미널에서 작동
- 코드 읽기/수정/실행, Git 관리에 최적화
- 내 컴퓨터 파일에 직접 접근 가능
- 비유: 개발자를 옆에 앉혀놓고 같이 코딩

**Cowork (Desktop 앱)**
- Claude Desktop 앱에서 작동
- 코딩뿐 아니라 모든 파일 작업 가능
- 폴더 지정 후 파일 읽기/만들기/정리 자동화
- 비유: 비서를 고용해서 "이 서류 정리해놔" 시키기

### 핵심 차이: 내 컴퓨터 파일 접근
```
claude.ai    → 내 컴퓨터 파일에 접근 못함 (대화만)
Claude Code  → 내 컴퓨터 파일에 접근 가능 (코드 중심)
Cowork       → 내 컴퓨터 파일에 접근 가능 (모든 파일)
```

### 역할 분담
```
claude.ai    → 개념 학습, 질의응답 (어디서든)
Claude Code  → 코드 생성/수정, Git 관리 (컴퓨터 앞에서)
Cowork       → 데이터 정리, 문서 작성, 파일 관리 (컴퓨터 앞에서)
```

### Claude Code vs Cowork
- 엔진은 같고 인터페이스가 다름
- Claude Code = 수동 기어 (세밀한 제어, 코딩에 최적)
- Cowork = 자동 기어 (편함, 일반 작업에 최적)
- 세 도구를 합치지 않는 이유: 보안 문제(브라우저에서 파일 접근), 편의성(터미널에서 문서 작업 불편)

### 핵심 질문과 이해

**Claude Code는 코딩만 가능한가?**
- Q: "Claude Code는 코딩에 대한 일들만 할 수 있어?"
- A: 코딩이 주력이지만 다른 일도 가능. Cowork가 만들어진 이유 자체가 사람들이 Claude Code를 코딩 외 일에 쓰기 시작해서.

**Cowork = claude.ai + Claude Code 합친 것?**
- Q: "Cowork는 결국 둘을 합쳐 놓은 느낌인데, 왜 다 따로 쓰지?"
- A: 접근 방식과 보안이 다름. claude.ai는 어디서든 접속 가능하지만 파일 접근 불가. Claude Code/Cowork는 파일 접근 가능하지만 로컬 앱/터미널 필요.

**Cowork로 코딩이 가능한가?**
- Q: "Cowork로 코딩을 하기는 조금 무리인가?"
- A: 가능하지만 Claude Code가 코딩에 더 최적화. 코드 실행, 프로젝트 구조 파악, Git 관리 등이 Claude Code에서 더 매끄러움.

### Cowork 첫 체험
- Claude Desktop 앱 설치 → Cowork 탭 진입
- Python 폴더 지정 후 작업 실행
- "이 폴더의 .py 파일들을 분석해서 요약 보고서를 만들어줘" → 파일 분석 → 폴더 생성 → 문서 작성 → 저장까지 자동 완료

## AI 트랙: 할루시네이션

### 할루시네이션이란
- AI가 그럴듯하지만 사실이 아닌 내용을 자신 있게 말하는 것
- 거짓말이 아니라, "다음에 올 확률이 높은 단어"를 예측하다 발생
- 일반 분야에서는 사소한 문제지만, 의료에서는 치명적

### 의료 할루시네이션의 위험
```
일반: "서울타워는 1975년에 지어졌습니다" (실제 1969년) → 별일 아님
의료: "이 약은 하루 3회 500mg 투여합니다" (최대 용량 초과) → 환자 위험!
```

### 95% 정확성의 역설
- AI가 완전히 엉터리면 오히려 안전 (바로 눈에 띄니까)
- 95%가 맞고 5%만 틀릴 때가 가장 위험
- 정확한 내용 속에 미묘한 오류가 섞이면 무심코 넘길 수 있음
- **신뢰가 쌓인 후의 오류가 가장 잡기 어려움**

### 실험 결과
- 와파린+NSAIDs 상호작용 질문 테스트
- AI 답변: 거의 정확하지만 사소한 실수 존재
- 느낀 점: "이 정도면 쓸만하다"

### 선생님 프로젝트에서의 대응 전략

**동의/반려 구조의 중요성**
```
AI 추천 5개 중:
  ✅ 1~4번 정확 → 신뢰가 쌓임
  ❌ 5번 미묘하게 틀림 → 1~4번이 맞으니까 넘길 수 있음
```
- AI를 100% 신뢰하지 않고, 의사가 하나씩 검토하는 구조 필수
- "똑똑한 레지던트의 제안을 어텐딩이 검토하는 구조"

**프롬프트 설계 원칙**
```
❌ "환자에게 필요한 처방을 알려줘"        → AI가 확정적으로 말함
✅ "필요할 수 있는 처방을 제안해줘.
   각 제안의 근거도 함께 알려줘"         → 근거를 보고 판단 가능
```
- AI에게 "확정"이 아니라 "제안 + 근거"를 요청
- 할루시네이션이 있어도 근거를 보고 잡아낼 수 있음
