### AI 에이전트 시대에 코딩 기초를 배우는 의미
- Q: "AI 에이전트가 내가 할 수 있는 일의 생산성을 훨씬 뛰어넘는데, 코딩을 기초부터 배우는 게 의미가 있을까?"
- A: AI 에이전트가 코드를 몇 초 만에 만들어주지만, 그 결과가 맞는지 판단하는 건 사람이 해야 함. 데이터 저장 구조가 적절한지, 에러 처리가 빠져있는지, 혈압 검증 로직이 의학적으로 맞는지 — AI는 물어보지 않으면 스스로 체크하지 않음.
- 방향 전환: "코드를 직접 짜는 것" → "AI가 만든 코드를 이해하고 지휘하는 것"으로 무게중심 이동
- 기초를 배워놓고 필요하면 그때 더 깊이 파면 됨

### 학습 방식 변경
```
기존: 개념 배우기 → 직접 코드 짜기 → 테스트
변경: 개념 배우기 → AI에게 시키기 → 결과 검토/수정 → 테스트
```

### Cowork와 일반 Claude의 차이
- Q: "일반 Claude와 Cowork의 차이를 잘 모르겠다"
- A: 일반 Claude는 "텍스트로 답변", Cowork는 "직접 컴퓨터를 조작"
  - 일반: "이 데이터를 분석해줘" → 분석 결과를 글로 써줌
  - Cowork: "이 데이터를 분석해줘" → 직접 파일을 열고, 코드를 짜서 실행하고, 결과를 엑셀로 만들어서 저장
- Q: "자동생성되는 AI 프롬프트 느낌인데?"
- A: 비슷하지만 Cowork 내부에서 AI가 스스로 코드를 짜서 실행하는 것. 코딩을 아는 사람이 결과를 검증할 수 있어야 함.

## 코딩 트랙: 모듈과 import

### 배운 개념

**모듈 분리 — 코드를 여러 파일로 나누기**
- 하나의 긴 파일을 역할별로 분리
- utils.py: 함수들 (도구 상자)
- day4_step3.py: 실행 코드 (사용자와 소통)

**import 방식 두 가지**
```python
# 방법 1: 필요한 것만 꺼내오기
from utils import 혈압판정, 환자브리핑
혈압판정(145, 92)              # 바로 사용

# 방법 2: 통째로 가져오기
import utils
utils.혈압판정(145, 92)        # "utils." 붙여야 함
```

### 핵심 질문과 이해

**from import vs import의 차이**
- Q: "json은 그냥 import json이라고 했는데, 왜 여기선 from utils import라고 하지?"
- A: json이 내장이라서가 아니라, 방식의 차이일 뿐. `import X`는 통째로 가져오기(X. 붙여야 함), `from X import Y`는 필요한 것만 꺼내오기(바로 사용 가능).

**왜 utils.을 붙여야 하는가**
- Q: "이미 함수가 불러와져 있는데 왜 utils.을 붙여야 하지?"
- A: 여러 파일에 같은 이름의 함수가 있을 수 있음. utils.저장하기()와 backup.저장하기()를 구분하기 위해. 내과 김 선생님, 외과 김 선생님 구분하는 것과 같음.

**import json이 utils.py에 있어도 되는 이유**
- Q: "실제 json 기능은 day4_step3.py에서 쓰잖아. import json이 utils.py에 있어야 하나?"
- A: json.dump(), os.path.exists()를 실제로 쓰는 곳이 utils.py 안의 함수들. day4_step3.py는 함수를 호출할 뿐, json을 직접 쓰지 않음.
- 비유: CT 기계(json)는 검사실(utils.py)에 있으면 됨. 진료실(day4_step3.py)에서는 의뢰(함수 호출)만 하면 됨.

**day4_step3.py의 역할**
- Q: "어차피 day4_step3.py는 함수가 불러온 json 형식을 출력만 해주는 거지?"
- A: 맞음. day4_step3.py는 함수를 호출하고 결과를 받아서 출력하는 역할. 복잡한 내부 작업은 utils.py에서 처리.

**import utils라고 해도 되는지**
- Q: "전체 기능을 다 불러오려면 import utils라고 해버려도 상관없나?"
- A: 가능. 대신 쓸 때 utils.혈압판정() 처럼 "utils."을 붙여야 함.

## Claude Code가 만든 코드에서 배운 새로운 개념

### .get(키, 기본값)
- 딕셔너리에서 값을 꺼내되, 키가 없으면 기본값을 쓰는 안전한 방법
- `진단별.get("고혈압", 0)` → 있으면 기존 값, 없으면 0
- `진단별["고혈압"]`으로 꺼내면 키가 없을 때 KeyError로 프로그램이 죽음

### 딕셔너리에 값 저장하는 원리 (심화 이해)
- Q: "진단별에는 고혈압:2 당뇨:1로 저장되는게 아니라 숫자만 저장되지 않나?"
- A: `진단별[진단] = 값`이라고 쓰면 키와 값이 함께 저장됨. 키를 지정하면서 넣는 것.
  - 키가 이미 있으면 → 값 교체
  - 키가 없으면 → 새로 추가

- Q: "수축기=130 대체와 고혈압:1 저장이 다른 개념 아닌가?"
- A: 딕셔너리는 항상 키:값 쌍으로 저장. 처음부터 그랬음. `환자["수축기"] = 130`도 "수축기"라는 키의 값을 바꾼 것.

- Q: "딕셔너리만 키:값 쌍으로 저장하는 건가?"
- A: 맞음. 딕셔너리 = 키:값 쌍, 리스트 = 순서대로 저장, 변수 = 이름 하나에 값 하나.

### != (같지 않다)
```python
==    # 같다
!=    # 같지 않다
```
- 초기 오해: "정상혈압인 경우를 세는 것"으로 이해
- 수정: `!=`이므로 "정상혈압이 아닌 경우"를 세는 것

### for 표현식 vs 일반 for문
- Q: "for 앞에 변수나 숫자가 오는 것이 일반 for문과 다른 느낌인데?"
- A: 둘은 다른 문법.
  - 일반 for문: 꺼내서 **동작 실행** (`for 환자 in 목록: print(...)`)
  - for 표현식: 꺼내서 **값 목록 생성** (`환자["나이"] for 환자 in 목록`)

- Q: "변수가 오는 것(환자["나이"])과 숫자가 오는 것(1)의 차이가 있다"
- A: 같은 원리. "각 환자마다 이 값을 내놓아라". 변수를 내놓으면 환자마다 다른 값, 숫자 1을 내놓으면 매번 같은 값. 1을 내놓는 이유는 개수를 세려고.

### sum() 함수
- 안에 있는 숫자를 전부 더하는 함수
- `sum(1, 1, 1) = 3` → 조건에 맞는 환자 3명

### .items()
- 딕셔너리에서 키와 값을 한 쌍씩 꺼내주는 기능
- `진단별.items()` → ("고혈압", 2), ("당뇨", 1) 순서로 꺼냄

### 언패킹 (unpacking) — 쉼표로 순서대로 나눠 받기
- Q: "for 진단, 수 에서 '수'라는 변수를 정의한 적이 없는데?"
- A: for문이 돌면서 그 자리에서 이름을 붙여주는 것 (enumerate와 같은 원리)

- Q: "콤마로 쓰면 자동으로 키와 값에 이름이 지정된다는 걸 몰랐다"
- A: Python의 "언패킹" 기능. 쉼표로 나열하면 순서대로 하나씩 받음. 첫 번째 = 키, 두 번째 = 값. for문뿐 아니라 어디서든 가능:
```python
이름, 나이 = "김철수", 58
```

- **순서가 의미를 결정함.** 뒤집어 쓰면 키가 값에, 값이 키에 들어감.

### :.1f (소수점 형식 지정)
- Q: "f가 두 번 들어가는데?"
- A: 첫 번째 f = f-string (값 채워넣기), `.1f`의 f = float (소수점 숫자 형식)
- `{평균나이:.1f}` → 소수점 첫째 자리까지만 표시
