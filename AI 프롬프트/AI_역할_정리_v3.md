# AI 역할 정리 (프로젝트 설계 기준)

## 1. 데이터 입력 시 — Free-text 분석 및 분배

### 기본 흐름
```
의사가 free-text로 차트 작성 (기존 습관 그대로)
    ↓
AI가 분석하여 각 테이블로 분배:
  - 진단 → 진단 테이블
  - 검사 수치 → 검사결과 테이블
  - 영상/특수검사 → 영상검사 테이블
  - 추적 계획 → 추적계획 테이블
  - 활력징후/생활습관 → 방문 테이블
  - 원본 free-text → 방문 테이블에 그대로 보존
    ↓
의사에게 분배 결과 보여주기 → 동의/반려 → 저장
```

### 진단명 파싱 — 핵심 진단명 + 부가 설명 분리
```
의사마다 쓰는 스타일이 다름:
  A 의사: "#고혈압(LSM)"
  B 의사: "#HTN - lifestyle modification"
  C 의사: "#고혈압 / 생활습관교정 중"
  D 의사: "#essential HTN, on LSM"

AI가 분배할 때 분리:
  → 진단명="고혈압" (표준화된 핵심 진단명)
  → 비고="LSM" (부가 설명)
  → 상태="활성"

  "#r/o 고혈압" → 진단명="고혈압", 비고="", 상태="의심"
  "#고혈압 의심" → 진단명="고혈압", 비고="", 상태="의심"

규칙:
  - 괄호, 쉼표, 대시 뒤의 내용 → 비고로 분리
  - r/o, 의심, rule out 등 → 상태를 "의심"으로 설정
  - 핵심 진단명은 통일된 형태로 저장 (통계용)
  - 의사 습관을 바꾸지 않음 — AI가 알아서 정리
```

### 진단 상태 — 3가지
```
활성   → 현재 해당되는 확진 진단 (#고혈압, #고혈압(LSM))
의심   → 아직 확진 안 됨 (#r/o 고혈압)
종결   → 더 이상 해당 안 됨 (오진, 완치 등)

상태 변경 흐름:
  의심 → 활성 (확진됨)
  의심 → 종결 (아닌 것으로 판명)
  활성 → 종결 (완치, 오진 등)
```

### 표준코드(KCD/ICD) 매핑
```
AI가 진단명을 분석할 때 가능하면 KCD/ICD 코드도 매핑:
  "고혈압" → 표준코드="I10"
  "제2형 당뇨" → 표준코드="E11"
  "경동맥 비후" → 표준코드=NULL (정확한 코드 없음)

저장:
  진단명="고혈압", 표준코드="I10"     ← 매핑 가능
  진단명="경동맥 비후", 표준코드=NULL  ← 매핑 불가

의사에게 확인:
  AI: "진단명='고혈압', KCD코드='I10'으로 매핑했습니다. 맞습니까?"
  → 동의/수정 후 저장
```

### 신규 vs 기존 데이터 구분
```
AI가 새 free-text를 받으면:
  1. 기존 검사결과 테이블과 비교
  2. 이미 저장된 수치와 같으면 → "기존 기록 참조" → 새로 저장 안 함
  3. 이미 저장된 수치와 다르면 → 의사에게 확인 요청:
     "기존 기록은 FBS 124인데 121로 쓰셨네요. 뭐가 맞나요?"
  4. 기존에 없는 수치면 → "새 검사결과" → 저장
```

### 날짜 자동 변환
```
의사가 "일주일 후 검사합시다"라고 쓰면:
  AI가 오늘 날짜 기준으로 계산 → "260122로 추적계획에 넣을까요?"
  의사 확인 후 저장
```

### 과거 검사결과 처리
```
의사가 "260324 LDL 165"라고 쓰면:
  AI가 날짜를 인식 → 검사시행일="260324"로 저장
  (오늘 방문일이 아닌 기록된 날짜로 저장)
```

### 검사결과 저장 원칙 — 항목별 개별 id 부여
```
같은 날 검사해도 항목마다 각각 저장:
  검사id=1, 260310, FBS, 108
  검사id=2, 260310, A1c, 5.8
  검사id=3, 260310, LDL, 162

이유: 연구 시 항목별 추적이 가능해야 하므로
  → "이 환자의 A1c 추이만 보기" = SQL 한 줄로 가능

화면에 보여줄 때는 같은 날짜끼리 묶어서 출력:
  260310  FBS 108, A1c 5.8, LDL 162
  (저장은 따로, 보여주기는 합쳐서)
```

### 외부 기록 사진 분석 (image_reader.py)
```
다른 병원 기록을 사진으로 촬영하여 입력:
  사진 촬영/업로드
      ↓
  AI가 이미지 분석 (OCR + 의미 파악)
      ↓
  JSON으로 정형화:
    {"진단": [...], "검사결과": [...], "처방": [...]}
      ↓
  의사 확인: "이렇게 넣을까요?"
      ↓
  동의 → 기존 DB 함수로 저장 (검사결과추가, 진단추가 등)

구현 원칙:
  - 기존 코드 수정 없음 — 새 모듈(image_reader.py)만 추가
  - from image_reader import 사진분석 → 한 줄로 연결
  - 추출된 데이터는 기존 DB 함수를 그대로 사용하여 저장
  - Claude API의 이미지 분석 기능 활용 (텍스트 분석과 코드 거의 동일)
```

## 2. 수정 시 — 데이터 유형별 다른 처리

### 검사결과/영상검사 (객관적 사실 — 원본 1개)

**중요: 같은 날 검사도 항목별로 각각 id가 부여됨**
```
DB 저장 (항목별 개별 id):
  검사id=1, 260310, FBS, 108
  검사id=2, 260310, A1c, 5.8
  검사id=3, 260310, LDL, 162

화면 출력 (같은 날짜끼리 묶어서):
  260310  FBS 108, A1c 5.8, LDL 162
```

**수정 시 — AI가 항목별로 비교하여 바뀐 것만 확인:**
```
의사가 free-text 수정: "260310 FBS 110, A1c 5.8, LDL 162"
                            ↑ 108→110 바뀜

AI가 항목별 비교:
  검사id=1, FBS: 기존 108 → 새 110 (변경 감지!)
  검사id=2, A1c: 기존 5.8 → 새 5.8 (동일 → 건너뜀)
  검사id=3, LDL: 기존 162 → 새 162 (동일 → 건너뜀)

AI: "FBS가 108→110으로 바뀌었는데 검사결과도 수정할까요?"
  → 승인 시 검사id=1만 수정 → 전체 자동 반영
  → 바뀌지 않은 항목은 건드리지 않음
```

### 진단 (시간에 따라 변함 — 이력 보존 필요)
```
진단 수정 시:
  → AI가 같은 진단명을 가진 모든 기록을 목록으로 보여줌:
    □ 진단id=1, 방문id=1 (260110 초진)
    □ 진단id=2, 방문id=2 (260210 경과)
    □ 진단id=3, 방문id=3 (260310 경과)
    ☑ 진단id=4, 방문id=4 (260415 약 시작)

  → 선택지 3가지:
    1. 이 차트만 수정 (1개)
    2. 선택한 차트들 수정 (직접 선택)
    3. 같은 진단 전부 수정

  → 예시: "고혈압"을 약 시작 전 차트만 "고혈압 의심"으로 바꾸고 싶을 때
    → 2번 선택 → 진단id 1,2,3 체크 → "고혈압 의심"으로 수정
    → 진단id=4(약 시작)는 "고혈압" 유지

진단 변경의 권장 방식:
  기존 진단 상태="종결" + 새 진단 추가 (이력 보존)
  예: "고혈압 의심"→종결 + "고혈압 확진" 새로 추가
```

### free-text 수정 시 전체 흐름
```
의사가 free-text 수정
  → AI가 수정 전후를 비교
  → 변경된 항목 감지:
    - 검사 수치 변경 → 항목별로 비교, 바뀐 것만 "수정할까요?" 확인
      (동일한 항목은 건너뜀, 변경된 항목만 확인)
    - 진단 변경 → "이 차트만? 선택한 차트들? 전부?" (3가지 선택지)
    - 추적계획 변경 → "추적계획 테이블도 수정할까요?"
    - 영상검사 변경 → 항목별로 비교, 바뀐 것만 확인
  → 각각 의사 확인 후 반영
  → 안 바뀐 항목은 자동 건너뜀 (물어보지 않음)
```

## 3. 삭제 시 — 데이터 보호 원칙

### 방문 기록 삭제
```
방문 삭제 시:
  → free_text만 삭제
  → 검사결과/영상검사: 절대 삭제 안 함 (방문과 독립)
  → 진단: 유지 (환자의 진단은 차트 하나와 무관)
  → 추적계획: 유지 (계획은 아직 유효할 수 있음)
```

### 진단 삭제
```
진단 삭제 시:
  → AI가 같은 진단명을 가진 모든 기록을 목록으로 보여줌
  → 선택지 3가지:
    1. 이 차트의 진단만 삭제 (1개)
    2. 선택한 차트들의 진단 삭제 (직접 선택)
    3. 같은 진단명 전부 삭제
  → 권장: 삭제보다 상태 변경 (활성→종결)
```

### 검사결과/영상검사 삭제
```
삭제 요청 시 AI가 같은 날짜의 항목 목록을 보여줌:

AI: "260310에 3건의 검사결과가 있습니다:
  □ 검사id=1, FBS 108
  □ 검사id=2, A1c 5.8
  □ 검사id=3, LDL 162

  1. 전부 삭제
  2. 선택해서 삭제"

→ 전부 선택 시: 3건 다 삭제
→ 선택 시: 원하는 항목만 체크해서 삭제
→ AI: "삭제하면 다른 차트에서도 참조할 수 없습니다. 진행할까요?"
→ 승인 시 삭제

영상검사도 동일한 방식:
AI: "260415에 1건의 영상검사가 있습니다:
  □ 영상id=1, 경초: Rt. IMT 1.1mm...
  삭제할까요?"
→ 승인 시 삭제
```

### 환자 전체 삭제
```
→ AI: "이 환자의 모든 기록(방문, 진단, 검사, 영상, 추적)이 삭제됩니다. 진행할까요?"
→ 승인 시 6개 테이블에서 해당 환자 기록 전부 삭제
```

## 4. 조회/브리핑 시 — 환자 이력 분석

### AI 브리핑 생성
```
의사가 환자를 선택하면:
  1. SQL로 해당 환자의 전체 데이터 조회 (JOIN)
  2. AI에게 전달: "이 환자의 이력을 요약해줘"
  3. AI가 브리핑 생성:
     - 현재 활성 진단 목록
     - 최근 검사 수치 추이
     - 미완료 추적계획
     - 주의사항
```

### 누락 체크
```
AI가 자동으로 확인:
  - "260112에 검사했는데 이후 방문에서 결과 확인 언급이 없습니다"
  - "추적계획 Lab(Lipid, LFT) 2605 예정인데 미완료입니다"
  - "마지막 BMD 검사 후 1년 경과했습니다"
```

### 차트 검토 (동의/반려 시스템)
```
의사가 free-text 작성 후:
  AI가 검토하여 추천 목록 제시:
    ✅ 1. HbA1c 검사 추천
    ✅ 2. 신장 기능 확인 필요
    ✅ 3. 복용 약물 상호작용 주의
    ✅ 4. 다음 방문 2주 후 권장
    ❌ 5. XX약 용량 증량 가능 (← AI가 틀릴 수 있음!)
  
  의사가 각 항목 동의/반려 → 반려 시 사유 기록
```

## 5. 연구 시 — 데이터 추출 및 분석

### 정형 데이터 검색 (SQL)
```
"60세 이상 고혈압 환자 중 LDL 160 이상인 사람"
  → AI가 SQL로 변환:
    SELECT ... FROM 환자 JOIN 검사결과 ...
    WHERE 나이 >= 60 AND 진단 = '고혈압' AND LDL >= 160
  → 바로 결과 반환
```

### 비정형 데이터 분석 (AI)
```
"두통을 호소한 환자를 찾아줘"
  → SQL로 free_text가 있는 방문 기록 조회
  → AI가 free_text를 읽고 "두통" 관련 기록 추출

"이 환자의 검사 목적을 정리해줘"
  → 검사결과에서 검사시행일 확인
  → 해당 날짜 전후 방문 free_text를 AI가 읽고 목적 추출
```

### 통계/분석
```
"A1c 추이를 분석해줘"
  → SQL: SELECT 검사시행일, 결과값 FROM 검사결과 WHERE 검사항목='A1c'
  → AI: "6.4(2508) → 6.1(2601) → 5.8(2604), 개선 추세"

"진단별 환자 수 통계"
  → SQL: SELECT 진단명, COUNT(*) FROM 진단 GROUP BY 진단명
```

### 표준코드 기반 연구 통계 (다중 사용자 환경)
```
연구 시 진단 통계를 돌릴 때 2단계로 처리:

1단계: 표준코드가 있는 진단 → SQL로 자동 포함 (확실)
  SELECT COUNT(*) FROM 진단 WHERE 표준코드 = 'I10'
  → 고혈압 20명

2단계: 표준코드가 NULL인 진단 → AI가 후보 추려서 의사에게 제시
  SELECT 진단명 FROM 진단 WHERE 표준코드 IS NULL
  → AI: "표준코드가 없는 진단 중 고혈압 관련 가능성:
    □ 혈압 경계치 (3명) — 포함할까요?
    □ 경동맥 비후 (2명) — 포함할까요?
    □ 검진 이상 (5명) — 관련 없어 보임"
  → 의사 판단: "혈압 경계치는 포함, 나머지는 제외"
  → 최종: 고혈압 23명

NULL은 빠지는 게 아니라 "따로 모아서 확인받는 것"
  코드 있음  = 확진 → 자동 포함
  코드 NULL = 리뷰 대상 → AI가 후보 추림 → 의사 판단 → 포함/제외
```

## 6. AI 안전 원칙

### 할루시네이션 대응
```
- AI에게 "확정"이 아니라 "제안 + 근거"를 요청
- 95% 정확해도 5%가 위험할 수 있음
- 신뢰가 쌓인 후의 오류가 가장 잡기 어려움
```

### 동의/반려 구조 (3중 안전장치)
```
1단계: AI가 기존 데이터와 비교하여 이상 감지
2단계: 의사에게 "이거 맞아요?" 확인 요청
3단계: 의사가 동의해야만 저장/수정/삭제
```

### Temperature 설정 (나중에 API 연동 시)
```
차트 검토:     temperature 0.0~0.2 (일관된 판단)
완성본 차트:   temperature 0.1~0.3 (의사 스타일 유지)
연구 데이터:   temperature 0.3~0.5 (다양한 시각 허용)
```

## 역할 분담 요약

```
SQL의 역할:
  - 정형 데이터 저장/검색/정렬/통계
  - 빠른 조건 검색 (실시간)
  - 데이터 무결성 보장 (FOREIGN KEY 등)

AI의 역할:
```
