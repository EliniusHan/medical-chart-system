## 1. chart_analyzer.py

### 1.1 _EXTRACTION_RULES — 공통 추출 규칙

**역할**: SYSTEM_PROMPT, REEXTRACT_SYSTEM 두 프롬프트가 공유하는 추출 규칙.

**핵심 원칙 (최우선)**:
- 결과 수치/소견이 차트에 적혀 있는가?
  - YES → lab_results 또는 imaging
  - NO  → test_orders (검사처방)

**의사 차트 스타일 규칙**:
```
1. 섹션 구분:
   * (1개)  → 섹션 구분자
   ** (2개) → 계획 항목 (날짜 유무로 '이미 한 것'과 '앞으로 할 것' 구분)

   [날짜 없는 **] → 방문일에 시행/처방한 것
     **약 N개월            → 처방 (일수=N×30, 오늘 처방) + 처방요약
     **Lab(검사명)         → 검사처방 "Lab (기본패널 + 괄호 안 항목)"
     **영상검사명 (CXR 등) → 검사처방 (결과 소견 없음)

   [날짜 있는 **] → 해당 월에 할 예정
     **YYMM Lab(검사명)    → 검사처방(YYMM00) + 추적계획(YYMM00)
     **YYMM 검사명          → 검사처방(YYMM00) + 추적계획(YYMM00)
     **YYMM 약 N개월       → 처방 + 추적계획

2. 하위 내용 표시: 줄 앞에 '-'가 붙으면 바로 윗줄의 하위 내용
3. 진단명 표기: #으로 시작, #r/o 또는 #의심이면 상태='의심', 괄호는 비고
4. 표준코드(KCD/ICD) 매핑
5. 생활습관 해석: 빈칸=물어보지 않음, '비흡연'/'없음'=확인함
6. 가족력/약부작용이력 감지 (변경유형: add/modify/remove)
7. 환자 테이블 정보와 차트 내용 불일치 시 → 환자 테이블 기준
```

**기존 데이터 비교**:
- 동일 → "구분": "기존참조" (저장 안 함)
- 다름 → "구분": "변경감지" + "기존값" 필드
- 신규 → "구분": "신규"

**날짜 변환 규칙**:
- YYMMDD (6자리) → 그대로 저장. 예: '260315'
- YYMM (4자리) → YYMM00으로. 예: '2603' → '260300'
- YYMM 4자리를 YYMMDD로 채우지 말 것! ('2603' → '260301' ❌)
- 00 = "이 달 중 언제인지 모름"

**검사결과 단위/참고범위 자동 채우기**:
40여 개 검사 항목 (CBC, LFT, 전해질, Lipid, FBS/A1c, TSH, UA 등)에 대해
단위와 참고범위를 AI 지식으로 자동 채움. 성별 반영.
저장 시에도 util.py의 _검사_참고정보 딕셔너리에서 한 번 더 채움 (이중 안전장치).

### 1.2 SYSTEM_PROMPT — 최초 분석

**역할**: 의사가 작성한 차트를 처음 분석 → 데이터 추출 + 검토 의견/법적 확인/설명의무 제안.

**출력 구조**:
```json
{
  "extraction": {
    "vitals": {"수축기": 152, ...},
    "lifestyle": {"흡연": "...", ...},
    "diagnoses": [{"진단명": "...", "상태": "활성", "구분": "신규"}],
    "lab_results": [... 수치 있는 것만],
    "imaging": [... 소견 있는 것만],
    "prescriptions": [...],
    "tracking": [...],
    "test_orders": [... 결과 없는 검사 전부],
    "prescription_summary": "...",
    "patient_info": {"가족력": {...}, "약부작용이력": {...}}
  },
  "suggestions": [
    {"icon": "💡", "content": "...", "reason": "가이드라인 근거",
     "chart_text": "차트에 삽입할 문구"}
  ],
  "legal": [
    {"icon": "⚖", "content": "...", "reason": "...", "chart_text": "..."}
  ],
  "informed_consent": {
    "drugs": [...], "side_effects": [...], "chart_text": "..."
  }
}
```

**검토 의견 (suggestions) 원칙**:
- 의사가 놓친 것, 추가로 고려할 것만 제안
- 이미 차트에 있는 내용 반복 금지
- 가이드라인 근거 명시 (ESC/ESH/ADA 등)
- DUR 금기, PubMed 논문, 급여 정보 포함
- 이전 방문 추세(악화/개선) 명시
- 각 제안에 차트 삽입 문구(chart_text) 포함

**법적 확인사항 (legal)**: 임신 가능성, 금연 권고, 간기능 확인 등

**설명의무 (informed_consent)**: 신규 처방 약물의 부작용 설명

### 1.4 _변경사항_추출 프롬프트 — free-text 수정 시

**역할**: 기존 free-text와 새 free-text를 비교하여 변경/추가/삭제된 데이터만 추출.

**시스템 프롬프트**:
```
당신은 의료 차트 데이터 비교 어시스턴트입니다.
수정 전후의 free-text를 비교하여 변경된 의료 데이터만 추출하세요.
새로운 제안이나 검토 의견은 절대 하지 마세요.
변경/추가/삭제된 데이터만 JSON으로 출력하세요.
```

**출력 JSON 형식**:
```json
{
  "변경": [
    {"테이블": "검사결과", "기존": {...}, "수정후": {...}, "검사시행일": "260301"}
  ],
  "추가": [
    {"테이블": "검사결과", "데이터": {"검사항목": "TG", "결과값": "150", ...}}
  ],
  "삭제": [
    {"테이블": "검사결과", "데이터": {...}}
  ],
  "활력징후_변경": {"수축기": 140, "이완기": 88} 또는 null,
  "처방요약_변경": "..." 또는 null
}
```

**추가 규칙**:
- 날짜 변환 규칙 (YYMM → YYMM00)
- 단위/참고범위 자동 채우기 (일반 검사 항목에 대해)

## 2. briefing_generator.py

### 2.1 SYSTEM_PROMPT — 전체 기록 브리핑

**역할**: 환자 전체 기록(8개 테이블)을 종합 분석하여 문제 중심 브리핑 생성.

**브리핑 형식**:
```
[진단/문제목록]
[현재 복용약]
[예정 검사]
[검사결과 추이]  ← 항목별 수치 변화, 화살표로 추이
[임상 요약]      ← 한 문단 서술형
[이번 방문 확인사항]  ← 진단/문제별 묶어서
[AI 권고사항]    ← 미시행 검사, 미이행 계획, 약물 모니터링 감지
```

**작성 원칙**:
- 각 진단별로 '왜 → 무엇을 → 어떻게'의 맥락
- 이전 소견 → 이번 조치 연결
- 마지막에 금일 핵심 업무 한 문장 요약
- 의학 약어 허용 (qd, f/u, Tx, Dx, LSM 등)

**시제 규칙 (매우 중요)**:
- 브리핑은 '오늘 환자를 보기 전' 읽는 문서
- 이전 기록 → '이전', '지난 방문', '기존' (과거형)
- 오늘 할 일 → '금일', '오늘', '이번 방문' (미래/현재형)
- 마지막 요약 → '오늘 OOO해서 OOO 확인해야 함' (행동 지시형)

잘못된 예:
```
- TSH 1.8로 정상범위 내 잘 조절됨  ← 이미 확인한 것처럼
- 4개월 후 TSH, FT4 추적 예정       ← 이미 결정한 것처럼
```

올바른 예:
```
- 레보티록신 75mcg qd 복용 중으로 이전 TSH 1.8로 잘 조절되고 있음  ← 이전
- 금일 혈액검사 시행하여 TSH, fT4 추적 예정                        ← 오늘
```

**AI 권고사항**:
1. 미시행 검사 감지 (진단별 가이드라인 기본 검사 누락)
2. 미이행 계획 감지 (추적계획 미완 / '예정'만 적고 실행 없음)
3. 약물 모니터링 감지 (스타틴→LFT, metformin→eGFR, ACEi→K/Cr 등)

### 2.2 SYSTEM_PROMPT_LATEST — 최근 차트만 브리핑

**역할**: 가장 최근 방문의 free-text 하나만 분석 → 브리핑 생성.

**사용 대상**: 매 차트에 이전 기록과 검사 결과를 요약해서 적는 스타일의 의사.

**브리핑 형식**:
```
[진단/문제목록]
[현재 복용약]
[검사결과 요약]
[임상 요약]
[이번 방문 확인사항]
[AI 권고사항]
```

**원칙**:
- 차트에 적힌 내용만 기반 (추측 금지)
- 데이터 없으면 '(차트에 미기재)' 표시
- 시제 규칙 동일

## 3. research_module.py

### 3.1 DB_SCHEMA — 공통 스키마 설명

SQL_SYSTEM, STAT_SYSTEM, STEP_SYSTEM이 공유하는 DB 구조 설명.

**주의사항**:
- 나이 계산: `(strftime('%Y','now') - substr(생년월일,1,4))`
- 결과수치(REAL) 칼럼 사용 (CAST 불필요)
- 진단 검색 시 상태='활성' 또는 '의심' 고려
- 날짜 끝이 00이면 월 단위 (예: 260300 = 2026년 3월 중)
- 유효여부 필터 규칙:
  - 환자 테이블에는 유효여부 칼럼 없음! (환자에는 유효여부 조건 금지)
  - 유효여부 있는 테이블: 방문, 진단, 검사결과, 영상검사, 추적계획, 처방, 검사처방

---

### 3.2 SQL_SYSTEM — 자연어 → SQL (NL2SQL)

**역할**: 의사의 자연어 질문을 SQLite SQL 쿼리로 변환.

**출력**: 실행 가능한 SQL만 (설명 없이).

**원칙**:
- JOIN으로 관련 테이블 연결
- 결과에 이름, 나이 등 기본정보 포함
- 유효여부=1 필터 규칙 준수

---

### 3.3 STAT_SYSTEM — AI 자동 통계 분석

**역할**: 의사의 연구 질문을 분석하여 SQL + 통계기법 + Python 코드 + 설명 생성.

**분석 유형 판단**:
1. **정형 데이터만으로 충분** (검사 수치, 진단명 등) → 표준 처리
2. **free_text 내용 분석 필요** (증상, 부작용 호소 등) → SQL로 후보 추리기 + free_text 분석

**free_text 분석 필요 예시**:
- "부작용을 호소한 환자" → 증상 표현 다양
- "금연 성공한 환자의 패턴"
- "환자 교육 내용 정리"

**불필요 예시**:
- "LDL 평균 구해줘"
- "고혈압 환자 수"

**출력 JSON (정형 데이터 분석)**:
```json
{
  "sql": "SELECT ...",
  "method": "paired t-test",
  "code": "import pandas as pd\nimport matplotlib...",
  "explanation": {
    "data_selection": "...",
    "grouping": "...",
    "method_reason": "...",
    "result_interpretation": "p-value=0.012로 유의..."
  }
}
```

**출력 JSON (free_text 분석)**:
```json
{
  "sql": "후보 환자 추림용 SQL",
  "analysis_type": "freetext",
  "freetext_query": "스타틴 관련 부작용 증상 찾아주세요...",
  "method": "freetext content analysis",
  "code": "",
  "explanation": {...}
}
```

**코드 작성 규칙**:
- 그래프 저장: `plt.savefig(graph_path, ...)` (graph_path는 자동 주입)
- plt.show() 금지, matplotlib.use('Agg') 필수
- DB 연결: `sqlite3.connect(r'{DB경로}')`
- 결과값 변환: `pd.to_numeric(..., errors='coerce')`
- explanation은 한국어, 논문 Methods/Results 수준
- result_interpretation은 실제 수치 기반, 플레이스홀더 금지

## 참고 사항

### 익명화 (anonymizer.py)
모든 AI 호출 전에 환자 정보 익명화 적용:
- 이름 → PT_XXXXX
- 생년월일 → 나이대 (50대 등)
- 병록번호 → MRN_XXXXX
- 환자id → 랜덤 숫자
- free_text, 처방요약 등 내부 텍스트도 치환
- **메모 필드는 AI로 전송 안 함** (선생님 사적 노트)

### 공공 DB 연동 (public_db.py)
chart_analyzer.py에서 처방 안전성 검토 시 자동 조회:
- DUR API (식약처): 병용금기, 임부금기, 연령금기
- e약은요 API (식약처): 효능, 부작용, 상호작용
- 약가마스터 (심평원): 급여 여부, 상한금액
- PubMed E-utilities: 관련 논문 근거
